🧠 Material para alumnos: Algoritmos y diagramas de flujo

1. Introducción

En programación, antes de escribir código, es importante pensar cómo resolver un problema. Muchas veces, los errores no aparecen por no saber un lenguaje como Python, sino por no haber organizado bien la solución.

Por eso, antes de programar, se trabaja con dos herramientas fundamentales:

  • el algoritmo, que describe los pasos para resolver un problema,
  • el diagrama de flujo, que representa esos pasos de forma visual.

En este material aprenderás qué son, para qué sirven y cómo se usan para diseñar una solución clara. Después aplicarás esas ideas a un problema concreto de bioinformática básica: analizar una secuencia de DNA y contar sus nucleótidos.

La idea principal es esta:

Primero se piensa la solución. Después se representa. Finalmente se programa.

2. ¿Qué es un problema computacional?

Un problema computacional es una situación que queremos resolver mediante una serie de pasos precisos.

Por ejemplo:

  • calcular el promedio de calificaciones,
  • ordenar una lista de números,
  • determinar si una palabra es palíndroma,
  • contar nucleótidos en una secuencia de DNA.

Cuando resolvemos un problema en programación, debemos identificar con claridad:

  • qué datos entran,
  • qué proceso se realiza,
  • qué resultado debe salir.

Esta forma de pensar ayuda a dividir el problema en partes más simples.

3. ¿Qué es un algoritmo?

Un algoritmo es una secuencia ordenada y finita de pasos que permite resolver un problema.

Características de un algoritmo

Un buen algoritmo debe ser:

  • ordenado: los pasos siguen una secuencia lógica,
  • claro: cada instrucción se entiende,
  • finito: debe terminar,
  • preciso: no debe ser ambiguo,
  • efectivo: sus pasos deben poder ejecutarse.

Ejemplo cotidiano

Preparar una bebida puede verse como un algoritmo:

  1. Tomar un vaso.
  2. Servir agua.
  3. Agregar polvo para bebida.
  4. Mezclar.
  5. Beber.

Esto muestra que un algoritmo no pertenece solo a la informática. Es una forma de organizar acciones para alcanzar un objetivo.

4. ¿Para qué sirve un algoritmo?

Trabajar con algoritmos permite:

  • organizar el pensamiento,
  • entender el problema antes de programar,
  • detectar errores lógicos,
  • comunicar la solución a otras personas,
  • convertir esa solución en código con mayor facilidad.

Un algoritmo es el puente entre el problema y el programa.

5. ¿Cómo se construye un algoritmo?

Para construir un algoritmo conviene seguir una estrategia de razonamiento.

Paso 1. Entender el problema

Preguntas útiles:

  • ¿Qué me están pidiendo resolver?
  • ¿Qué datos voy a recibir?
  • ¿Qué resultado debo obtener?
  • ¿Qué restricciones existen?

Paso 2. Identificar entrada, proceso y salida

Una manera simple de analizar cualquier problema es dividirlo en tres partes:

  • Entrada: los datos que recibe el programa.
  • Proceso: las operaciones que se realizan con esos datos.
  • Salida: el resultado que se muestra o se devuelve.

Paso 3. Descomponer el problema

Un problema grande casi siempre puede dividirse en tareas pequeñas.

Por ejemplo, si queremos contar nucleótidos en DNA, podemos separar el trabajo en:

  • leer la secuencia,
  • convertirla a mayúsculas,
  • validar que sea correcta,
  • contar cada nucleótido,
  • mostrar resultados.

Paso 4. Ordenar los pasos

No basta con saber qué hacer. También hay que decidir en qué orden hacerlo.

Paso 5. Revisar condiciones

Muchos problemas incluyen decisiones como:

  • si el dato está vacío,
  • si un valor es inválido,
  • si se cumple o no cierta condición.

Estas decisiones son fundamentales en el diseño del algoritmo.

6. ¿Qué es un diagrama de flujo?

Un diagrama de flujo es una representación gráfica de un algoritmo.

En lugar de escribir los pasos solo con texto, se usan símbolos conectados con flechas para mostrar:

  • el orden de ejecución,
  • las acciones del proceso,
  • las decisiones que se toman,
  • las entradas y salidas.

Los diagramas de flujo ayudan a visualizar mejor la lógica del problema.

7. ¿Para qué sirven los diagramas de flujo?

Los diagramas de flujo son útiles porque permiten:

  • comprender mejor el algoritmo,
  • identificar errores de lógica,
  • comunicar la solución de manera visual,
  • planear un programa antes de escribir código,
  • relacionar condiciones, caminos y resultados.

En otras palabras, un diagrama de flujo permite ver cómo piensa el algoritmo.

8. Símbolos básicos de un diagrama de flujo

8.1 Inicio / Fin

Se usa para indicar dónde comienza y termina el algoritmo.

Forma habitual: óvalo.

Ejemplos:

  • Inicio
  • Fin

8.2 Proceso

Representa una acción o instrucción.

Forma habitual: rectángulo.

Ejemplos:

  • Leer secuencia
  • Convertir a mayúsculas
  • Contar nucleótidos

8.3 Entrada / Salida

Representa datos que entran o resultados que salen.

Forma habitual: paralelogramo.

Ejemplos:

  • Ingresar secuencia
  • Mostrar resultado

8.4 Decisión

Representa una pregunta cuya respuesta suele ser sí o no.

Forma habitual: rombo.

Ejemplos:

  • ¿La secuencia está vacía?
  • ¿Los caracteres son válidos?

8.5 Flechas

Indican la dirección del flujo del algoritmo.

Permiten seguir el recorrido lógico paso a paso.

9. Cómo leer un diagrama de flujo

Para leer un diagrama de flujo correctamente, debes:

  1. localizar el inicio,
  2. seguir las flechas,
  3. leer cada proceso en orden,
  4. observar las decisiones y sus posibles caminos,
  5. continuar hasta llegar al fin.

Cuando aparece una decisión, el algoritmo puede tomar rutas distintas según la respuesta.

Guía breve para dibujar diagramas de flujo (con Mermaid)

🧱 Elementos básicos

Símbolo Uso Ejemplo Mermaid

10. Relación entre algoritmo, pseudocódigo y programa

Es importante distinguir estos tres niveles:

Algoritmo

Es la idea general de la solución.

Pseudocódigo

Es una forma escrita del algoritmo usando lenguaje natural estructurado. No depende de un lenguaje de programación específico.

Ejemplo:

leer secuencia
convertir a mayúsculas
si la secuencia está vacía
    mostrar error
si no
    validar caracteres
    contar nucleótidos
    mostrar resultados

Programa

Es la implementación del algoritmo en un lenguaje como Python.

La secuencia didáctica ideal es:

problema → algoritmo → diagrama/pseudocódigo → programa

11. Estrategia para razonar antes de programar

Cuando enfrentes un problema, puedes usar estas preguntas guía:

Sobre la entrada

  • ¿Qué dato recibe el programa?
  • ¿El usuario escribe texto, números o una secuencia?
  • ¿Puede haber errores en la entrada?

Sobre el proceso

  • ¿Qué debo hacer con el dato?
  • ¿Necesito contar, comparar, convertir o validar?
  • ¿Hay decisiones que deban tomarse?

Sobre la salida

  • ¿Qué debe mostrar el programa?
  • ¿Cómo debe verse el resultado?
  • ¿Qué pasa si el dato es incorrecto?

Estas preguntas ayudan a construir la solución con lógica, no por adivinación.

12. Aplicación al problema del DNA

12.1 Enunciado del problema

Se desea crear un programa que lea una secuencia de DNA ingresada por teclado y reporte el conteo de nucleótidos:

  • A
  • T
  • C
  • G

12.2 Qué debe hacer el programa

El programa debe:

  1. pedir una secuencia,
  2. convertirla a mayúsculas,
  3. revisar si está vacía,
  4. verificar que solo contenga A, T, C y G,
  5. contar cuántas veces aparece cada nucleótido,
  6. mostrar el resultado.

13. Análisis del problema con entrada, proceso y salida

Entrada

Una secuencia de DNA escrita por el usuario.

Ejemplo:

ATGCGCATTA

Proceso

  • normalizar la secuencia,
  • validar caracteres,
  • contar nucleótidos,
  • calcular el total.

Salida

Un reporte como este:

A: 3
T: 3
C: 2
G: 2
Total: 10

14. Descomposición del problema

Resolver este problema es más sencillo si lo dividimos en partes pequeñas:

Parte 1. Leer la secuencia

Se obtiene la información del usuario.

Parte 2. Normalizar

Se convierte la secuencia a mayúsculas para trabajar de forma uniforme.

Parte 3. Validar

Se verifica que la secuencia no esté vacía y que no contenga símbolos inválidos.

Parte 4. Contar

Se recorre la secuencia y se incrementa el contador correspondiente a cada nucleótido.

Parte 5. Mostrar resultados

Se imprime el conteo final de A, T, C y G, además del total.

15. Algoritmo propuesto para este problema

Inicio

1. Pedir al usuario una secuencia de DNA.
2. Leer la secuencia.
3. Convertir la secuencia a mayúsculas.
4. Verificar si la secuencia está vacía.
5. Si está vacía, mostrar un mensaje de error y terminar.
6. Revisar cada carácter de la secuencia.
7. Si aparece un carácter diferente de A, T, C o G, mostrar un mensaje de error y terminar.
8. Inicializar contadores para A, T, C y G.
9. Recorrer la secuencia.
10. Incrementar el contador correspondiente según el nucleótido leído.
11. Calcular el total de nucleótidos.
12. Mostrar los resultados.

Fin

16. Diagrama de flujo del problema

flowchart TD
    A([Inicio]) --> B[/Leer secuencia de DNA/]
    B --> C[Convertir a mayúsculas]
    C --> D{¿La secuencia está vacía?}
    D -->|Sí| E[/Mostrar mensaje de error/]
    E --> F([Fin])
    D -->|No| G[Revisar caracteres]
    G --> H{¿Todos los caracteres son válidos?}
    H -->|No| I[/Mostrar mensaje de error/]
    I --> F
    H -->|Sí| J[Inicializar contadores]
    J --> K[Contar A, T, C y G]
    K --> L[Calcular total]
    L --> M[/Mostrar resultados/]
    M --> F

17. Cómo razonar este diagrama

Observa que el diagrama no empieza contando inmediatamente. Antes de procesar la secuencia, el algoritmo debe comprobar si la entrada es válida.

Esto enseña una idea muy importante:

No todo problema se resuelve directamente. Primero hay que verificar si los datos permiten continuar.

El flujo muestra dos decisiones esenciales:

  • si la secuencia está vacía,
  • si la secuencia contiene caracteres válidos.

Solo cuando ambas condiciones se cumplen, el algoritmo pasa al conteo.

18. Errores comunes al diseñar algoritmos

Al comenzar, es normal cometer algunos errores frecuentes:

  • querer programar antes de entender el problema,
  • no distinguir entre entrada, proceso y salida,
  • olvidar casos de error,
  • no considerar datos vacíos,
  • escribir pasos ambiguos,
  • omitir decisiones importantes,
  • hacer diagramas sin seguir una secuencia lógica.

Detectar estos errores es parte del aprendizaje.

19. Recomendaciones para resolver problemas

Cuando diseñes una solución, recuerda:

  • lee el enunciado con atención,
  • subraya qué entra y qué debe salir,
  • divide el problema en pasos pequeños,
  • identifica decisiones,
  • usa pseudocódigo si todavía no estás listo para programar,
  • revisa si el algoritmo termina,
  • comprueba si contempla errores o casos límite.

20. Casos que conviene pensar antes de programar

Para este problema, algunos casos importantes son:

  • una secuencia válida como ATGC,
  • una secuencia en minúsculas como atgc,
  • una secuencia vacía,
  • una secuencia con caracteres inválidos como ATBX,
  • una secuencia de un solo nucleótido,
  • una secuencia con muchas repeticiones.

Pensar estos casos ayuda a mejorar el algoritmo antes de escribir código.

21. Idea central que debes recordar

La programación no comienza cuando escribes código. Comienza cuando entiendes el problema y organizas la solución.

Por eso, antes de usar Python, debes aprender a:

  • analizar el problema,
  • construir un algoritmo,
  • representarlo con un diagrama de flujo,
  • revisar su lógica.

Cuando haces eso, programar se vuelve mucho más fácil.

A continuación se proponen actividades con ejemplos concretos para que practiques el razonamiento algorítmico usando contextos sencillos de bioinformática.

Actividad 1 — Identificar entrada, proceso y salida

Ejemplo 1: Transcripción de DNA a RNA

Regla: T → U

  • Entrada: ATGCT
  • Proceso: reemplazar cada T por U
  • Salida: AUGCU

Ejemplo 2: Longitud de una secuencia

  • Entrada: ATGCGC
  • Proceso: contar número de caracteres
  • Salida: 6

Actividad 2 — ¿Qué es un algoritmo? (con ejemplo)

🔹 Algoritmo (explicado en lenguaje natural)

Un algoritmo describe la idea de la solución, sin preocuparse todavía por la sintaxis.

1. Pedir al usuario una secuencia de DNA.
2. Convertir la secuencia a mayúsculas.
3. Verificar si la secuencia está vacía.
4. Si está vacía, indicar error y terminar.
5. Revisar cada carácter de la secuencia.
6. Si algún carácter no es A, T, C o G, indicar error y terminar.
7. Si todos los caracteres son válidos, indicar que la secuencia es correcta.

👉 Observa que: - Está escrito como una lista de pasos. - No depende de ningún lenguaje de programación. - Usa lenguaje natural claro.

Ahora tomamos el mismo algoritmo y lo expresamos en pseudocódigo.

🔹 Pseudocódigo

leer secuencia
convertir a mayúsculas

si la secuencia está vacía
    mostrar "Error: secuencia vacía"
si no
    para cada carácter en la secuencia
        si el carácter no es A, T, C o G
            mostrar "Error: carácter inválido"
            terminar
    mostrar "Secuencia válida"

🔍 Diferencia clave entre algoritmo y pseudocódigo

Aspecto Algoritmo Pseudocódigo
Nivel Conceptual Intermedio (cercano al código)
Lenguaje Natural Estructurado
Sintaxis Libre Similar a programación
Objetivo Entender la solución Preparar para programar

💡 Idea importante

El pseudocódigo es un paso intermedio entre el algoritmo y el programa.

Problema → Algoritmo → Pseudocódigo → Programa

23. Cierre

Aprender algoritmos y diagramas de flujo es aprender a pensar soluciones de manera ordenada.

En este tema, el objetivo no es memorizar símbolos, sino desarrollar una forma de razonamiento que después puedas llevar a cualquier programa.

Hoy trabajamos con una secuencia de DNA, pero esta misma forma de análisis sirve para muchísimos problemas de programación.

Un buen programa empieza con una buena idea organizada paso a paso.
