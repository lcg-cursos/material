6. ¿Qué es un módulo?

Un módulo en Python es un archivo .py que contiene código que puede ser reutilizado.

Ejemplos:

filters.py
core.py
exporters.py
cli.py

Todos esos archivos pueden ser módulos.

Algo muy importante

Un módulo no es algo especial ni misterioso.

Un módulo es simplemente un archivo .py que Python puede importar.

8. Modularización mínima: crear filters.py

Empezaremos moviendo las funciones de filtrado.

¿Por qué empezar con filtros?

Porque son funciones con una responsabilidad clara:

  • reciben un regulón
  • aplican un criterio
  • devuelven un regulón filtrado

Además, no escriben archivos ni leen argumentos de línea de comandos.

Eso las hace buenas candidatas para estar juntas.

Paso 1: crear el archivo

Desde la raíz del proyecto ejecuta:

touch src/filters.py

Tu proyecto debe quedar así:

src/
├── regulon_summary.py
└── filters.py

Paso 2: mover funciones a filters.py

Corta desde regulon_summary.py y pega en filters.py estas funciones:

def get_regulator_type(data):
    """
    Determina el tipo de regulador.
    """
    activados = data["activados"]
    reprimidos = data["reprimidos"]

    if activados > 0 and reprimidos > 0:
        return "dual"
    elif activados > 0:
        return "activador"
    else:
        return "represor"
def filter_by_min_genes(regulon, min_genes):
    """
    Filtra un regulón por número mínimo de genes regulados.
    """
    filtered = {}

    for TF, data in regulon.items():
        if len(data["genes"]) >= min_genes:
            filtered[TF] = data

    return filtered
def filter_by_type(regulon, regulator_type):
    """
    Filtra reguladores por tipo de regulación.
    """
    if regulator_type is None:
        return regulon

    filtered = {}

    for TF, data in regulon.items():
        current_type = get_regulator_type(data)

        if current_type == regulator_type:
            filtered[TF] = data

    return filtered

¿Por qué también movemos get_regulator_type()?

Porque filter_by_type() la necesita.

Si filter_by_type() queda en filters.py, pero get_regulator_type() queda en regulon_summary.py, tendríamos una dependencia incómoda.

Por ahora, moverlas juntas simplifica la práctica.

Más adelante podríamos mover get_regulator_type() a un módulo llamado core.py, porque representa lógica biológica central.

Paso 3: importar en regulon_summary.py

En la parte superior de src/regulon_summary.py, agrega:

from filters import filter_by_min_genes, filter_by_type, get_regulator_type

Esto significa:

Desde el módulo filters.py, trae estas tres funciones para poder usarlas aquí.

Paso 4: ejecutar de nuevo

Desde la raíz del proyecto:

uv run python src/regulon_summary.py data/raw/NetworkRegulatorGene.tsv results/regulon_summary.tsv

Si todo funciona, hiciste una refactorización correcta.

9. ¿Qué es refactorizar?

Refactorizar significa cambiar la organización interna del código sin cambiar su comportamiento externo.

Antes:

regulon_summary.py contenía todo

Después:

regulon_summary.py coordina
filters.py contiene filtros

Pero el comando para usar el programa sigue siendo el mismo.

Tip

Refactoriza en pasos pequeños:

mover poco → ejecutar → verificar → continuar

No muevas muchas funciones al mismo tiempo sin probar.

10. Modularización completa del proyecto

La modularización mínima ya funciona, pero podemos organizar mejor el proyecto.

La estructura final sugerida para esta práctica es:

src/
├── regulon_summary.py
├── io_utils.py
├── core.py
├── filters.py
├── exporters.py
└── cli.py

Usamos io_utils.py en lugar de io.py para evitar confusión con el módulo estándar io de Python.

¿Qué va en cada archivo?

Archivo Funciones sugeridas Razón
io_utils.py load_interactions() Lectura de datos de entrada
core.py build_regulon(), get_regulator_type() Lógica biológica central
filters.py filter_by_min_genes(), filter_by_type() Criterios de selección
exporters.py write_summary(), write_sif() Escritura de salidas
cli.py parse_arguments() Argumentos de línea de comandos
regulon_summary.py main() Coordina el flujo completo

Orden recomendado para mover funciones

No muevas todo de golpe.

Sigue este orden:

  1. filters.py
  2. core.py
  3. exporters.py
  4. io_utils.py
  5. cli.py
  6. Dejar regulon_summary.py solo con main() y los imports necesarios

Después de cada paso, ejecuta el programa.

Paso D: crear cli.py

    ...

Resultado esperado final

regulon_summary.py debería quedar como un archivo que coordina el pipeline:

from cli import parse_arguments
from core import build_regulon
from exporters import write_summary, write_sif
from filters import filter_by_min_genes, filter_by_type
from io_utils import load_interactions


def main():
    args = parse_arguments()

    input_file = args.input_file
    output_file = args.output_file
    min_genes = args.min_genes
    regulator_type = args.type
    output_format = args.format

    interactions = load_interactions(input_file)
    regulon = build_regulon(interactions)
    regulon = filter_by_min_genes(regulon, min_genes)
    regulon = filter_by_type(regulon, regulator_type)

    if output_format == "summary":
        write_summary(regulon, output_file)
    elif output_format == "sif":
        write_sif(interactions, output_file)


if __name__ == "__main__":
    main()

Puede que tu versión tenga validaciones y try/except. No las elimines. Solo asegúrate de que main() siga coordinando el flujo.

12. Debugging básico: cómo leer errores

Cuando algo falle, no borres cambios al azar.

Lee el error.

ModuleNotFoundError

Ejemplo:

ModuleNotFoundError: No module named 'filters'

Significa:

Python no encontró el archivo/módulo que intentaste importar.

Revisa:

  • ¿Existe src/filters.py?
  • ¿Escribiste bien el nombre?
  • ¿Estás ejecutando desde la raíz del proyecto?

ImportError

Ejemplo:

ImportError: cannot import name 'filter_by_min_gene'

Significa:

Python encontró el módulo, pero no encontró esa función dentro del módulo.

Revisa:

  • ¿La función existe?
  • ¿Está bien escrito el nombre?
  • ¿Moviste la función al archivo correcto?

NameError

Ejemplo:

NameError: name 'get_regulator_type' is not defined

Significa:

Estás usando un nombre que Python no conoce en ese archivo.

Revisa:

  • ¿Falta un import?
  • ¿La función fue movida a otro archivo?
  • ¿Cambió el nombre?

FileNotFoundError

Ejemplo:

FileNotFoundError: data/raw/NetworkRegulatorGene.tsv

Significa:

Python no encontró el archivo de datos en la ruta indicada.

Revisa:

  • ¿Estás en la raíz del proyecto?
  • ¿Existe el archivo?
  • ¿La ruta está bien escrita?

14. ¿Qué es __name__? Experimento paso a paso

Esta sección es importante. Hazla con calma.

__name__ es una variable especial que Python crea automáticamente en cada archivo .py.

Su valor depende de cómo se usa el archivo.

Experimento 1: archivo ejecutado directamente

Paso 1: crear un archivo nuevo

Desde la raíz del proyecto:

touch src/ver_name.py

Paso 2: escribir este código en src/ver_name.py

print("Estoy en ver_name.py")
print("El valor de __name__ es:", __name__)

Paso 3: ejecutarlo directamente

Desde la raíz del proyecto:

uv run python src/ver_name.py

Salida esperada

Estoy en ver_name.py
El valor de __name__ es: __main__

Explicación

Cuando ejecutas:

python src/ver_name.py

ese archivo es el punto de entrada del programa.

Python lo considera el programa principal.

Por eso asigna:

__name__ = "__main__"

Experimento 2: el mismo archivo importado desde otro archivo

Ahora veremos que el mismo archivo puede comportarse distinto.

Paso 1: crear otro archivo

Desde la raíz:

touch src/importar_ver_name.py

Paso 2: escribir este código en src/importar_ver_name.py

print("Estoy en importar_ver_name.py")

import ver_name

print("Terminé de importar ver_name")

Paso 3: ejecutar importar_ver_name.py

Desde la raíz:

uv run python src/importar_ver_name.py

Salida esperada

Estoy en importar_ver_name.py
Estoy en ver_name.py
El valor de __name__ es: ver_name
Terminé de importar ver_name

Explicación detallada

El programa principal ahora es:

importar_ver_name.py

Ese archivo tiene __name__ = "__main__" porque fue ejecutado directamente.

Pero ver_name.py no fue ejecutado directamente. Fue importado.

Por eso, dentro de ver_name.py:

__name__ = "ver_name"

Python usa el nombre del módulo, no "__main__".

Conclusión del experimento

El valor de __name__ depende del contexto:

| Situación | Valor de __name__ | | | | | Archivo ejecutado directamente | "__main__" | | Archivo importado como módulo | nombre del módulo |

15. ¿Por qué importa esto?

Porque en Python, cuando importas un archivo, Python ejecuta el contenido del archivo.

Ejemplo:

# ejemplo.py
print("Esto se ejecuta al importar")

Si otro archivo hace:

import ejemplo

el mensaje se imprime.

Esto puede ser un problema si dentro del archivo hay código que lee datos, escribe archivos o ejecuta un análisis completo.

16. El problema de llamar main() sin protección

Supón que tienes:

def main():
    print("Ejecutando análisis")

main()

Si ejecutas el archivo directamente, funciona.

Pero si lo importas desde otro archivo, también se ejecuta main() automáticamente.

Eso puede causar problemas.

En bioinformática podría significar:

  • Leer archivos grandes sin querer.
  • Sobrescribir resultados.
  • Ejecutar análisis largos accidentalmente.
  • Fallar porque no se pasaron argumentos de línea de comandos.

17. La solución: if __name__ == "__main__"

La forma correcta es:

def main():
    print("Ejecutando análisis")


if __name__ == "__main__":
    main()

Esto significa:

Ejecuta main() solo si este archivo fue ejecutado directamente.

Si el archivo se importa, main() no se ejecuta automáticamente.

Experimento 3: proteger main()

Paso 1: crear archivo

touch src/demo_main.py

Paso 2: escribir en src/demo_main.py

def main():
    print("Ejecutando main() en demo_main.py")


if __name__ == "__main__":
    main()

Paso 3: ejecutar directamente

uv run python src/demo_main.py

Salida esperada:

Ejecutando main() en demo_main.py

Paso 4: crear archivo que lo importe

touch src/importar_demo_main.py

Dentro de src/importar_demo_main.py:

import demo_main

print("demo_main fue importado, pero main() no se ejecutó automáticamente")

Paso 5: ejecutar el importador

uv run python src/importar_demo_main.py

Salida esperada:

demo_main fue importado, pero main() no se ejecutó automáticamente

Explicación

Cuando demo_main.py se importa, su __name__ vale:

"demo_main"

Entonces esta condición es falsa:

__name__ == "__main__"

Por eso no se llama a main().

19. Tips importantes sobre módulos

Tip 1

Si una función tiene una responsabilidad clara y podría usarse en otro lugar, puede vivir en un módulo.

Tip 2

No pongas código ejecutable suelto en módulos reutilizables.

Evita esto:

print("Cargando módulo")

O esto:

resultado = load_interactions("archivo.tsv")

porque se ejecutará al importar.

Tip 3

Después de mover una función, busca dónde se usaba y agrega el import necesario.

Tip 4

Si aparece un NameError, probablemente olvidaste importar algo.

Tip 5

Si aparece ModuleNotFoundError, revisa ubicación, nombre del archivo y desde dónde ejecutas.

  • Refactoriza
  • Ejecuta pytest después

👉 Si las pruebas pasan, tu cambio no rompió funcionalidad.

24. Introducción a test-driven thinking

En esta práctica no vamos a hacer TDD formal, pero sí vamos a introducir una forma de pensar muy útil:

Antes de cambiar código, piensa qué comportamiento esperas comprobar.

Esto se llama aquí test-driven thinking.

La idea no es escribir pruebas por obligación, sino usar las pruebas como una forma de razonar mejor sobre el programa.

¿Qué significa pensar guiado por pruebas?

Significa que antes de modificar una función te preguntas:

  1. ¿Qué entrada voy a darle?
  2. ¿Qué salida espero?
  3. ¿Qué caso simple debería funcionar?
  4. ¿Qué caso límite podría fallar?
  5. ¿Cómo sabré que no rompí algo que ya funcionaba?

Ejemplo con filter_by_min_genes()

Antes de tocar la función, podemos pensar:

Si tengo un regulón con dos TFs, uno con 2 genes y otro con 1 gen, y filtro con min_genes=2, debería conservar solo el primero.

Eso se traduce a una prueba:

def test_filter_by_min_genes():
    regulon = {
        "CRP": {"genes": ["lacZ", "cyaA"], "activados": 1, "reprimidos": 1},
        "FNR": {"genes": ["narG"], "activados": 0, "reprimidos": 1},
    }

    result = filter_by_min_genes(regulon, 2)

    assert "CRP" in result
    assert "FNR" not in result

Observa la estructura mental

Una prueba tiene tres partes:

1. Preparar datos
2. Ejecutar la función
3. Verificar el resultado

En inglés se conoce como:

Arrange → Act → Assert

En español podemos pensarlo como:

Preparar → Ejecutar → Comprobar

Ejemplo con filter_by_type()

Antes de probar, razonamos:

Si CRP activa y reprime, entonces es dual.
Si FNR solo reprime, entonces es represor.
Al filtrar por dual, debería quedar solo CRP.

Prueba:

def test_filter_by_type_dual():
    regulon = {
        "CRP": {"genes": ["lacZ", "cyaA"], "activados": 1, "reprimidos": 1},
        "FNR": {"genes": ["narG"], "activados": 0, "reprimidos": 1},
    }

    result = filter_by_type(regulon, "dual")

    assert "CRP" in result
    assert "FNR" not in result

¿Por qué esto ayuda al programar?

Porque evita programar “a ciegas”.

Sin prueba:

Creo que funciona.

Con prueba:

Tengo un ejemplo concreto que demuestra que funciona para este caso.

Mini-ciclo recomendado

Cuando agregues o modifiques una función, sigue este ciclo:

1. Escribe un ejemplo pequeño
2. Escribe la prueba
3. Ejecuta pytest
4. Observa si falla
5. Ajusta el código
6. Ejecuta pytest otra vez

Actividad: romper una prueba a propósito

Para entender el valor de las pruebas, haremos un error intencional.

En filter_by_min_genes(), cambia temporalmente:

if len(data["genes"]) >= min_genes:

por:

if len(data["genes"]) > min_genes:

Ahora ejecuta:

PYTHONPATH=src uv run pytest

La prueba debería fallar.

¿Por qué falla?

Porque si min_genes=2, un TF con exactamente 2 genes debería conservarse.

Con > ya no se conserva.

La prueba detecta que cambiamos el comportamiento esperado.

Después de observar el error

Regresa la línea correcta:

if len(data["genes"]) >= min_genes:

Ejecuta de nuevo:

PYTHONPATH=src uv run pytest

Ahora las pruebas deberían pasar otra vez.

Idea clave

Una buena prueba no solo verifica que el código funciona.
También documenta lo que esperamos que el código haga.

Diferencia entre probar manualmente y usar pytest

| Prueba manual | Prueba con pytest | | | | | Ejecutas comandos y revisas resultados | Ejecutas pruebas automatizadas | | Depende de que recuerdes qué revisar | La prueba ya sabe qué comprobar | | Útil al inicio | Útil para mantener el proyecto | | Difícil de repetir muchas veces | Fácil de repetir |

Test-driven thinking en este proyecto

Antes de agregar una nueva función, por ejemplo:

def filter_by_regulator_name(...):
    ...

Primero deberías poder decir:

Dado este regulón pequeño, si filtro por CRP, espero obtener solo CRP.

Luego escribes la prueba.

Después escribes o ajustas la función.

Preguntas de reflexión

  1. ¿Qué caso concreto prueba test_filter_by_min_genes()?
  2. ¿Qué comportamiento documenta test_filter_by_type_dual()?
  3. ¿Por qué fue útil romper una prueba intencionalmente?
  4. ¿Qué prueba agregarías para un regulador represor?
  5. ¿Qué prueba agregarías para el caso regulator_type=None?

25. Cierre conceptual

Un script resuelve una tarea.

Un programa modular puede crecer.

Las pruebas permiten confiar en ese crecimiento.

Pensar primero en ejemplos esperados mejora la forma en que diseñamos funciones.
