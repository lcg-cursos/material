🔬 Lección completa: Pruebas de software con regulon_summary

1. Propósito de esta lección

Hasta ahora has construido un programa que ya resuelve un problema real: leer interacciones regulatorias desde un archivo, procesarlas y generar un resumen por factor de transcripción.

Ese programa no es un ejercicio aislado. Ya tiene varias características de software real:

  • recibe entradas desde línea de comandos
  • lee archivos
  • transforma datos
  • escribe resultados
  • valida condiciones de error

Eso significa que ya llegaste a un punto muy importante del desarrollo:

ya no solo necesitas que el programa funcione una vez; necesitas poder comprobar, de manera confiable, que sigue funcionando cuando lo modificas.

Ese es el objetivo de esta lección.

Aquí aprenderás a introducir pruebas automatizadas usando como base el programa:

src/regulon_summary.py

La meta no es memorizar comandos ni aprender testing como tema aislado. La meta es entender por qué las pruebas aparecen de forma natural cuando un programa empieza a crecer.

2. El problema que queremos resolver

Imagina esta situación.

Ya terminaste una primera versión funcional de regulon_summary. El programa corre, produce un archivo de salida y aparentemente todo está bien.

Pero después aparece una nueva necesidad:

  • quieres evitar contar genes repetidos
  • quieres mejorar el filtrado por número mínimo de genes
  • quieres cambiar parte de la salida
  • quieres reorganizar funciones para que el código quede más claro

En ese momento aparece una pregunta muy importante:

¿Cómo sabrás que tus cambios no rompieron algo que antes sí funcionaba?

Puedes correr el programa a mano una vez. Incluso dos o tres veces. Pero eso rápidamente deja de ser suficiente.

Por ejemplo:

  • quizás pruebas solo un archivo y no otro
  • quizás olvidas revisar un caso límite
  • quizás el programa sigue corriendo, pero ahora cuenta mal
  • quizás el mensaje de error cambió sin darte cuenta

Ese es el problema que resuelven las pruebas.

Las pruebas no existen porque “así se hace profesionalmente” y ya.

Existen porque, cuando un programa empieza a tener más de una parte importante, necesitas una manera sistemática de verificar su comportamiento.

3. Por qué esta lección sigue después de manejo de errores

Esta secuencia no es casualidad.

Antes trabajaste:

  • estructuras de control
  • estructuras de datos
  • funciones
  • lectura y escritura de archivos
  • argumentos por línea de comandos
  • manejo de errores

Y además ya definiste casos de prueba en:

docs/casos_prueba.md

Eso significa que ya hiciste algo fundamental: pensaste en distintos escenarios.

Por ejemplo:

  • qué pasa si el archivo no existe
  • qué pasa si una línea está incompleta
  • qué pasa si el archivo está vacío
  • qué pasa si el filtro deja al resultado sin TFs

Eso ya es pensamiento de pruebas.

La diferencia es que hasta ahora esos escenarios estaban escritos como documentación o se verificaban manualmente.

El siguiente paso lógico es este:

convertir esos casos en verificaciones que se puedan ejecutar automáticamente.

Por eso esta lección sigue naturalmente después de manejo de errores.

Porque cuando aprendes a anticipar fallos, también empiezas a pensar cómo verificarlos.

4. Qué son las pruebas automatizadas

Una prueba automatizada es una forma de expresar en código algo como esto:

Dada esta entrada, el programa o una parte del programa debería comportarse de esta manera.

Una prueba no consiste en “correr el programa y ver si parece estar bien”.

Una prueba consiste en definir de manera explícita:

  1. qué situación quieres evaluar
  2. qué parte del programa vas a ejecutar
  3. qué resultado esperas observar

En términos simples, una prueba tiene esta estructura:

preparar datos → ejecutar → verificar

Por ejemplo:

  • preparas una lista pequeña de interacciones
  • llamas a build_regulon()
  • verificas que el diccionario resultante tenga los conteos correctos

O bien:

  • intentas abrir un archivo que no existe
  • llamas a load_interactions()
  • verificas que se produzca la excepción esperada

Eso ya es una prueba.

5. Qué NO son las pruebas automatizadas

Es importante aclarar algo.

Las pruebas automatizadas no sustituyen el razonamiento.

No sirven de mucho si primero no entendiste qué debería hacer el programa.

Tampoco son simplemente “usar pytest”.

pytest es una herramienta. Pero la parte más importante ocurre antes: pensar bien el caso.

Por eso, en esta lección, no vamos a empezar escribiendo tests de inmediato.

Primero vamos a revisar los casos de prueba que ya existen y a convertirlos en algo que realmente pueda verificarse en código.

6. Punto de partida: docs/casos_prueba.md

Tu proyecto ya cuenta con un documento de casos de prueba:

docs/casos_prueba.md

Eso es muy valioso, porque significa que ya no partimos de una hoja en blanco.

En ese documento ya existen escenarios importantes del programa.

Sin embargo, esos casos suelen estar expresados en lenguaje natural. Por ejemplo:

  • “si el archivo no existe, debe mostrar error”
  • “si el efecto es +-, cuenta como activado y reprimido”
  • “si una línea está incompleta, debe ignorarse”

Ese tipo de redacción sirve muy bien para pensar y comunicar el comportamiento esperado. Pero todavía no está lista para convertirse directamente en una prueba automatizada.

¿Por qué?

Porque una prueba necesita mayor precisión.

Necesita responder preguntas como:

  • ¿qué función específica voy a probar?
  • ¿qué dato concreto voy a usar como entrada?
  • ¿qué resultado observable voy a verificar?
  • ¿espero un valor, un archivo generado o una excepción?

7. Refinar casos de prueba: del lenguaje natural a lo verificable

Aquí está una de las ideas más importantes de toda la lección.

No vamos a tirar ni rehacer los casos de prueba que ya tienes.

Vamos a refinarlos.

Es decir:

  • conservar el comportamiento que describen
  • volverlo más preciso
  • hacerlo comprobable en código

Ejemplo 1

Caso original:

Si el archivo no existe, debe mostrar error.

Versión refinada:

  • Función a probar: load_interactions
  • Entrada: "no_existe.tsv"
  • Resultado esperado: se lanza FileNotFoundError

Ahora el caso ya se puede convertir en una prueba real.

Ejemplo 2

Caso original:

Si el efecto es +-, debe contar como activado y reprimido.

Versión refinada:

  • Función a probar: build_regulon
  • Entrada:
[("CRP", "lacZ", "+-")]
  • Resultado esperado:
    • activados = 1
    • reprimidos = 1

Ejemplo 3

Caso original:

Si una línea tiene pocas columnas, debe ignorarse.

Versión refinada:

  • Función a probar: load_interactions
  • Entrada: archivo temporal con una línea incompleta
  • Resultado esperado: la línea no aparece en la lista de interacciones devuelta

Observa la diferencia.

En los tres casos, el comportamiento sigue siendo el mismo. Lo que cambió es el nivel de precisión.

8. Actividad 1: refinar casos de prueba

Antes de escribir tests, vas a trabajar sobre los casos existentes.

Objetivo

Transformar casos de prueba en lenguaje natural en casos verificables.

Instrucciones paso a paso

  1. Abre el archivo:
docs/casos_prueba.md

  1. Elige al menos 3 casos de prueba.

  2. Para cada uno, escribe una versión refinada con estos elementos:

  • nombre del caso
  • función a probar
  • entrada concreta

  • resultado esperado verificable

  1. Agrega una nueva sección al final del documento, por ejemplo:
## Casos adaptados a testing
  1. Coloca ahí tus versiones refinadas.

¿Qué significa “entrada concreta”?

No basta con escribir algo como:

  • “archivo incorrecto”
  • “datos inválidos”
  • “caso simple”

Debes escribir algo que pueda usarse realmente en una prueba. Por ejemplo:

  • filename = "no_existe.tsv"
  • una lista pequeña de tuplas
  • un archivo temporal con dos líneas válidas y una inválida

¿Qué significa “resultado verificable”?

Debe ser algo que se pueda comprobar con claridad, por ejemplo:

  • un valor exacto
  • una clave presente en un diccionario
  • una longitud esperada
  • una excepción específica
  • una línea escrita en un archivo

Frases como estas no son suficientemente verificables:

  • “debe funcionar correctamente”
  • “debe verse bien”
  • “debe procesarse sin problema”

9. Uso de IA para refinar casos

Puedes usar IA como apoyo en esta etapa, pero hay una condición importante:

la IA no debe decidir por ti qué comportamiento importa; tú ya debiste pensarlo antes.

Un uso correcto de IA aquí es pedir ayuda para volver más precisos los casos.

Puedes usar este prompt:

Estoy trabajando en un programa en Python llamado regulon_summary.
Tengo casos de prueba escritos en lenguaje natural en docs/casos_prueba.md.
Ayúdame a convertirlos en casos verificables para pytest.

No escribas código todavía.
Para cada caso dame:
- nombre del caso
- función a probar
- entrada concreta
- resultado esperado verificable

No inventes nuevos requisitos.
Solo transforma los existentes.

Qué debes revisar después de usar IA

Después de obtener una propuesta, revisa:

  • si la función elegida tiene sentido
  • si la entrada realmente es concreta
  • si el resultado esperado se puede verificar
  • si la IA no cambió el significado del caso original

10. Qué parte del programa conviene probar primero

Aunque el programa completo puede probarse, no conviene empezar por la parte más compleja.

Lo mejor es comenzar por funciones pequeñas o relativamente aisladas.

En este proyecto, una secuencia razonable es:

  1. build_regulon(interactions)
  2. load_interactions(filename)
  3. write_summary(regulon, output_file)
  4. más adelante, si hace falta, el flujo de main()

Veamos por qué.

10.1 build_regulon

Es una muy buena función para empezar porque:

  • recibe datos en memoria
  • no necesita leer archivos
  • no depende de argumentos CLI
  • devuelve una estructura que puedes inspeccionar fácilmente

Eso la vuelve ideal para una primera prueba.

10.2 load_interactions

Esta función ya implica archivos, así que es un poco más compleja.

Pero sigue siendo muy buena para practicar porque permite probar:

  • líneas vacías
  • comentarios
  • filas incompletas
  • efectos inválidos
  • archivo inexistente

10.3 write_summary

Aquí practicas otra idea importante: verificar salida generada.

No solo se prueban valores en memoria. También se puede probar:

  • si el archivo se creó
  • si el encabezado está presente
  • si el contenido tiene el formato esperado

11. La herramienta que vamos a usar: pytest

Para automatizar pruebas vamos a usar pytest.

pytest es una herramienta muy popular en Python porque permite escribir pruebas con una sintaxis simple y legible.

En muchos casos, una prueba básica se ve así:

def test_algo():
    assert resultado == esperado

Y cuando esperas un error, puedes escribir algo como esto:

import pytest

with pytest.raises(FileNotFoundError):
    load_interactions("no_existe.tsv")

Eso hace que la herramienta no estorbe: el foco sigue estando en entender el comportamiento del programa.

12. Preparación del proyecto para pruebas

Paso 1: agregar pytest

Como el proyecto usa uv, puedes agregar pytest como dependencia de desarrollo con:

uv add --dev pytest

¿Qué significa “dependencia de desarrollo”?

Que se usa para trabajar sobre el proyecto, pero no forma parte de la lógica principal del programa.

En este caso, pytest se usa para verificar el software, no para ejecutar regulon_summary en producción.

Paso 2: crear carpeta tests/

En la raíz del proyecto crea una carpeta llamada:

tests/

Una estructura sencilla puede quedar así:

regulatory-network/
├── docs/
│   └── casos_prueba.md
├── src/
│   └── regulon_summary.py
├── tests/
│   ├── test_build_regulon.py
│   ├── test_load_interactions.py
│   └── test_write_summary.py
└── pyproject.toml

Esta separación ayuda a distinguir claramente entre:

  • el código del programa
  • el código que verifica al programa

13. Cómo se ejecutan las pruebas

Una vez instalado pytest, puedes ejecutar las pruebas con:

uv run pytest -v

¿Qué hace ese comando?

  • uv run ejecuta el comando dentro del entorno del proyecto
  • pytest busca y corre las pruebas
  • -v significa “verbose”, es decir, muestra más detalle

Cuando todo esté bien, verás que las pruebas pasan.

Si algo falla, pytest te mostrará qué test falló y en qué línea.

Eso también es importante: una prueba no solo sirve cuando pasa. También sirve cuando falla, porque te ayuda a ubicar el problema.

14. Primera estrategia: probar build_regulon

Vamos a empezar con la función más accesible.

¿Por qué empezar aquí?

Porque permite concentrarte en la idea central de una prueba sin distraerte con archivos o CLI.

Caso sencillo

Supón que tienes estas interacciones:

interactions = [
    ("AraC", "araA", "+"),
    ("AraC", "araB", "-"),
    ("LexA", "recA", "-")
]

Si build_regulon() funciona bien, tendriamos una salida como:

{
    "AraC": {
        "genes": {"araA", "araB"},
        "activados": {"araA"},
        "reprimidos": {"araB"}
    },
    "LexA": {
        "genes": {"recA"},
        "activados": set(),
        "reprimidos": {"recA"}
    }
}

entonces deberíamos poder verificar cosas como estas:

Que los TF esperados existen

Esto verifica que build_regulon() creó entradas para ambos factores de transcripción.

assert "AraC" in regulon
assert "LexA" in regulon

Que los genes únicos se guardaron bien

assert regulon["AraC"]["genes"] == {"araA", "araB"}
assert regulon["LexA"]["genes"] == {"recA"}

Aquí estamos comprobando que:

  • AraC regula dos genes únicos
  • LexA regula un solo gen único

Que los genes activados se registraron correctamente

assert regulon["AraC"]["activados"] == {"araA"}
assert regulon["LexA"]["activados"] == set()
  • AraC activa araA
  • LexA no activa ningún gen en este caso

Que los genes reprimidos se registraron correctamente

assert regulon["AraC"]["reprimidos"] == {"araB"}
assert regulon["LexA"]["reprimidos"] == {"recA"}

Esto verifica que:

  • AraC reprime araB
  • LexA reprime recA

También podríamos verificar tamaños

assert len(regulon["AraC"]["genes"]) == 2
assert len(regulon["AraC"]["activados"]) == 1
assert len(regulon["AraC"]["reprimidos"]) == 1

assert len(regulon["LexA"]["genes"]) == 1
assert len(regulon["LexA"]["activados"]) == 0
assert len(regulon["LexA"]["reprimidos"]) == 1

Esto es útil cuando quieres enfatizar el conteo, no necesariamente los nombres exactos.

15. Live coding 1: primer test real

En esta parte vas a escribir tu primera prueba automatizada.

Paso 1: crear el archivo de pruebas

Crea un archivo:

tests/test_build_regulon.py

Paso 2: importar la función

Dependiendo de cómo esté organizado tu proyecto, importarás la función desde src/regulon_summary.py.

Paso 3: escribir el test

Un primer ejemplo puede verse así:

from src.regulon_summary import build_regulon

def test_build_regulon_basico():
    interactions = [
        ("AraC", "araA", "+"),
        ("AraC", "araB", "-"),
        ("LexA", "recA", "-")
    ]

    regulon = build_regulon(interactions)

    assert "AraC" in regulon
    assert "LexA" in regulon

    assert regulon["AraC"]["genes"] == {"araA", "araB"}
    assert regulon["AraC"]["activados"] == {"araA"}
    assert regulon["AraC"]["reprimidos"] == {"araB"}

    assert regulon["LexA"]["genes"] == {"recA"}
    assert regulon["LexA"]["activados"] == set()
    assert regulon["LexA"]["reprimidos"] == {"recA"}

Qué está haciendo este test

Este test sigue exactamente la lógica de una prueba:

  1. prepara una entrada pequeña y controlada
  2. ejecuta la función
  3. verifica el resultado

¿Por qué la entrada debe ser pequeña?

Porque cuando estás aprendiendo a probar, conviene que el caso sea lo más claro posible.

Si el ejemplo tiene demasiados datos, se vuelve más difícil entender qué está verificando el test.

16. Qué debes observar durante el live coding

Mientras escribes y ejecutas esta primera prueba, detente a pensar en estas preguntas:

  • ¿qué comportamiento específico estoy comprobando?
  • si la lógica interna cambiara, ¿este test detectaría el error?
  • ¿el nombre del test describe bien lo que verifica?
  • ¿la prueba está revisando algo importante o algo superficial?

Estas preguntas son tan importantes como el código mismo.

17. Live coding 2: probar un caso especial

Ahora puedes probar el caso del efecto +-.

Ejemplo

from src.regulon_summary import build_regulon


def test_build_regulon_effect_dual_counts_both():
    interactions = [
        ("CRP", "lacZ", "+-")
    ]

    rregulon = build_regulon(interactions)

    assert "CRP" in regulon

    assert regulon["CRP"]["genes"] == {"lacZ"}
    assert regulon["CRP"]["activados"] == {"lacZ"}
    assert regulon["CRP"]["reprimidos"] == {"lacZ"}

Qué aporta este test

Este caso ya no revisa lo más básico. Revisa una regla particular del problema.

Eso es importante porque muestra que una prueba puede capturar no solo el funcionamiento general, sino también detalles específicos del comportamiento esperado.

¿Qué pasaría si hay duplicados?

("CRP", "lacZ", "+-"),
("CRP", "lacZ", "+-")
assert regulon["CRP"]["genes"] == {"lacZ"}

18. Segunda estrategia: probar load_interactions

Ahora vamos a una función que depende de archivos.

Aquí el objetivo es verificar que la función lea correctamente solo las líneas válidas y descarte las que no cumplen las condiciones.

Qué tipo de comportamientos puedes probar

  • ignorar líneas vacías
  • ignorar comentarios
  • ignorar encabezado
  • ignorar líneas con pocas columnas
  • ignorar efectos inválidos
  • devolver solo interacciones válidas

Además, esta función permite probar errores de lectura, por ejemplo:

  • archivo inexistente

19. Cómo probar lectura de archivos sin depender de un archivo real del proyecto

En testing, no siempre conviene usar un archivo real de tu carpeta data/.

¿Por qué?

Porque para una prueba, lo ideal es construir un escenario pequeño, controlado y fácil de entender.

Para eso, pytest ofrece recursos como tmp_path, que permite crear archivos temporales dentro del test.

Eso te ayuda a escribir pruebas más limpias y más independientes.

20. Live coding 3: probar lectura básica con tmp_path

Ejemplo:

from src.regulon_summary import load_interactions


def test_load_interactions_reads_only_valid_lines(tmp_path):
    input_file = tmp_path/"test.tsv"
    input_file.write_text(
        "# comentario\n"
        "1)regulatorId\tregulatorName\tgeneId\tgeneName\tsite\tevidence\teffect\n"
        "id1\tAraC\tg1\taraA\ts1\tev1\t+\n"
        "id2\tAraC\tg2\taraB\ts2\tev2\t-\n"
        "incompleta\tAraC\n"
        "id3\tCRP\tg3\tlacZ\ts3\tev3\tx\n"
    )

    interactions = load_interactions(str(input_file))

    assert len(interactions) == 2
    assert interactions[0] == ("AraC", "araA", "+")
    assert interactions[1] == ("AraC", "araB", "-")

Qué verifica este test

Verifica varias cosas a la vez:

  • que se ignoren comentarios
  • que se ignore el encabezado
  • que se ignore la línea incompleta
  • que se ignore una línea con efecto inválido
  • que solo se conserven las interacciones válidas

¿Está bien verificar varias cosas en un solo test?

A veces sí, si todas forman parte del mismo comportamiento general.

Sin embargo, cuando un test empieza a cubrir demasiadas ideas distintas, puede volverse difícil de entender.

Por eso, más adelante, podrías dividir este tipo de pruebas en casos más pequeños.

21. Live coding 4: probar error esperado

También debemos probar que la función falle cuando corresponde.

Por ejemplo, si el archivo no existe:

import pytest
from src.regulon_summary import load_interactions


def test_load_interactions_raises_error_if_file_does_not_exist():
    # Ejecuta el código dentro de este bloque y verifica que ocurra esta excepción.
    with pytest.raises(FileNotFoundError):
        load_interactions("no_existe.tsv")

Por qué este test es importante

Porque muestra algo fundamental:

las pruebas no son solo para confirmar casos exitosos; también sirven para verificar que los errores correctos ocurran en el momento correcto.

Esto conecta directamente con la lección anterior sobre manejo de errores.

22. Tercera estrategia: probar write_summary

La función write_summary() permite practicar otra idea muy importante: verificar salida generada por el programa.

Aquí el objetivo ya no es solo revisar valores en memoria, sino comprobar si el archivo de salida realmente contiene lo esperado.

Cosas que puedes verificar

  • que el archivo se cree
  • que el encabezado se escriba
  • que aparezcan los TF correctos
  • que las columnas contengan los conteos esperados
  • que el tipo (activador, represor, dual) se determine correctamente

23. Live coding 5: probar escritura básica

Ejemplo:

from src.regulon_summary import write_summary


def test_write_summary_creates_output_file(tmp_path):
    regulon = {
        "AraC": {
            "genes": {"araA", "araB"},
            "activados": {"araA"},
            "reprimidos": {"araB"}
        }
    }

    output_file = tmp_path/"summary.tsv"

    write_summary(regulon, str(output_file))

    # Verificaciones
    assert output_file.exists()

    content = output_file.read_text()

    assert "AraC" in content
    assert "araA" in content
    assert "araB" in content

Qué se puede mejorar después

Este test es una primera verificación.

Más adelante puedes volverlo más estricto revisando:

  • el encabezado exacto
  • el número exacto de líneas
  • el contenido exacto de la fila de AraC

Eso muestra algo importante: las pruebas también pueden mejorar con el tiempo.

24. Cómo pensar buenos tests

A medida que escribas pruebas, conviene seguir algunas reglas sencillas.

1. Mantén cada test enfocado

Un test debe tener una intención clara.

Cuando alguien lo lea, debería poder responder rápidamente:

“¿Qué comportamiento verifica este test?”

2. Usa datos pequeños

Entre más pequeño sea el ejemplo, más fácil será entenderlo y detectar qué está fallando.

3. Usa nombres claros

El nombre del test debe describir el comportamiento esperado.

Por ejemplo:

  • test_build_regulon_basico
  • test_load_interactions_raises_error_if_file_does_not_exist

4. Verifica cosas importantes

No llenes los tests de assert superficiales.

Verifica lo que realmente importa del comportamiento.

5. No inventes comportamiento nuevo en el test

El test debe corresponder a un requisito o a un caso de prueba existente.

Si el test impone una regla que nunca fue definida, puede convertirse en una fuente de confusión.

25. Uso de IA para generar tests

Una vez que ya refinaste tus casos de prueba, puedes usar IA para ayudarte a convertirlos en código pytest.

Pero, otra vez, la regla es la misma:

la IA ayuda a escribir; tú decides si el test tiene sentido.

Puedes usar un prompt como este:

Estoy trabajando en un proyecto en Python llamado regulon_summary.
Ya tengo casos de prueba refinados con:
- función a probar
- entrada concreta
- resultado esperado verificable

Ayúdame a convertirlos en tests con pytest.

Reglas:
- no modifiques el programa
- no inventes requisitos nuevos
- usa assert o pytest.raises según corresponda
- usa tmp_path si se requieren archivos temporales
- mantén cada test corto y claro

Qué debes revisar después

  • si el test realmente corresponde al caso refinado
  • si el assert verifica algo importante
  • si la IA no agregó complejidad innecesaria
  • si el test fallaría de verdad cuando el comportamiento es incorrecto

26. Qué hacer cuando un test falla

Cuando una prueba falla, no significa automáticamente que el programa esté mal.

Tampoco significa automáticamente que el test esté bien.

Cuando un test falla, debes preguntarte:

  1. ¿el caso está bien planteado?
  2. ¿la entrada corresponde a lo que quiero evaluar?
  3. ¿el resultado esperado es correcto?
  4. ¿el error está en el programa o en la prueba?

Este punto es importante porque enseña una forma más madura de trabajar.

No se trata solo de “hacer que la prueba pase”.

Se trata de entender qué está diciendo la prueba sobre el comportamiento del software.

27. Relación entre pruebas y diseño del programa

Las pruebas también te enseñan algo sobre la calidad del diseño.

Cuando una función es difícil de probar, muchas veces eso revela que está haciendo demasiadas cosas al mismo tiempo.

Por ejemplo, si toda la lógica del programa estuviera dentro de main(), probar sería mucho más complicado.

En cambio, cuando el programa está dividido en funciones como:

  • load_interactions
  • build_regulon
  • write_summary

entonces las pruebas salen de manera más natural.

Esto refuerza una idea muy importante del curso:

el código bien organizado no solo se lee mejor; también se verifica mejor.

29. Resumen conceptual de la lección

En esta lección no aprendiste solo a usar pytest.

Aprendiste algo más importante:

  • que los casos de prueba primero se piensan
  • que después pueden refinarse
  • que luego pueden convertirse en pruebas ejecutables
  • que también se prueban errores, no solo casos exitosos
  • que las pruebas ayudan a modificar el programa con más seguridad

La secuencia completa queda así:

Requisitos → Casos de prueba → Implementación → Manejo de errores → Pruebas automatizadas

Esto muestra que el testing no es un tema aislado. Es una continuación natural de todo lo que ya venías haciendo.

30. Mensaje final

Las pruebas no son un lujo ni una tarea extra puesta al final del curso.

Son una herramienta para pensar mejor el programa.

Un programa puede parecer correcto cuando lo ejecutas una vez.

Pero cuando el proyecto crece, cuando haces cambios, cuando otras personas leen tu código o cuando quieres mejorarlo sin romperlo, las pruebas se vuelven parte del trabajo normal de desarrollo.

En otras palabras:

un programa sin pruebas puede funcionar hoy; un programa con pruebas puede evolucionar mañana.
