Lección: Organización del código con funciones

1. Evolución del software y necesidad de organización del código

Contexto

En las primeras versiones de un programa:

  • el problema es pequeño
  • el código es corto
  • una solución lineal suele ser suficiente

En este curso, el programa fue creciendo por etapas:

  • primero trabajamos con datos en memoria
  • después incorporamos lectura desde archivo
  • luego agregamos validaciones
  • finalmente generamos un archivo de salida

Hasta este punto, el programa ya resuelve el problema y cumple los requisitos funcionales definidos.

Importancia de los requisitos

Una buena práctica en desarrollo de software es:

identificar lo mejor posible los requisitos desde el inicio

Esto ayuda a:

  • reducir cambios posteriores
  • evitar retrabajo
  • facilitar el diseño
  • disminuir errores

Sin embargo, en la práctica real:

los programas cambian constantemente

Porque:

  • aparecen nuevos casos de uso
  • se detectan errores
  • se solicitan mejoras
  • cambian las necesidades del usuario

Por eso, además de entender los requisitos, también necesitamos aprender a mantener la calidad del código cuando el programa crece.

Problema: crecimiento del código

Con cada cambio, el programa:

  • crece en tamaño
  • mezcla responsabilidades
  • se vuelve más difícil de leer
  • se vuelve más difícil de modificar
  • aumenta el riesgo de romper algo que ya funcionaba

En este punto, el problema ya no es solo resolver el dominio biológico.

Ahora también necesitamos resolver un problema de diseño:

cómo organizar el código para que siga siendo claro, mantenible y fácil de extender

Refactorización

Una técnica fundamental para enfrentar este problema es la refactorización.

Refactorizar significa reorganizar y mejorar el código sin cambiar su comportamiento observable

En otras palabras:

  • mismo input
  • mismo output
  • mejor estructura interna

Objetivos de la refactorización

  • mejorar legibilidad
  • facilitar mantenimiento
  • reducir errores
  • preparar el programa para cambios futuros

Regla clave

Después de refactorizar:

el programa debe seguir produciendo exactamente los mismos resultados

Por eso, es indispensable apoyarse en casos de prueba.

Rol de las funciones

Una de las herramientas más importantes para refactorizar es el uso de funciones.

Las funciones permiten:

  • dividir el programa en partes claras
  • asignar una responsabilidad a cada parte
  • reutilizar lógica
  • probar componentes por separado
  • hacer más entendible el flujo del programa

Objetivo de esta lección

En esta sesión:

  • no vamos a cambiar la funcionalidad del programa
  • vamos a reorganizarlo usando funciones

No vamos a escribir un programa nuevo, sino aprender a transformar un script lineal en un programa con mejor estructura.

2. Del código lineal al código organizado

El programa inicial

Partimos de un script como este:

import os

# =========================================
# Lectura del archivo y construcción de interactions
# =========================================

interactions = []

filename = "data/raw/NetworkRegulatorGene.tsv"

if not os.path.exists(filename):
    print("Error: archivo no encontrado")
    exit(1)
else:
    with open(filename) as f:        
        for line in f:
            
            line = line.strip()

            # Ignorar líneas vacías
            if not line:
                continue

            # Ignorar comentarios
            if line.startswith("#"):
                continue

            # Ignorar encabezado
            if line.startswith("1)regulatorId"):
                continue
            
            fields = line.split("\t")

            # Validar número mínimo de columnas
            if len(fields) <= 6:
                continue
                        
            TF = fields[1]
            gene = fields[4]
            effect = fields[5]

            # Validar effect
            if effect not in ["+", "-","+-"]:
                continue

            interactions.append((TF, gene, effect))


# =========================================
# Construcción del regulon con información extra
# =========================================

regulon = {}

for TF, gene, effect in interactions:
    
    # Inicializar estructura si el TF no existe
    if TF not in regulon:
        regulon[TF] = {
            "genes": [],
            "activados": 0,
            "reprimidos": 0
        }

    # Agregar gen a la lista
    regulon[TF]["genes"].append(gene)

    # Contar tipo de regulación
    if effect == "+":
        regulon[TF]["activados"] += 1
    elif effect == "-":
        regulon[TF]["reprimidos"] += 1
    elif effect == "+-":
        regulon[TF]["activados"] += 1
        regulon[TF]["reprimidos"] += 1  


# =========================================
# Generación de la salida
# =========================================

output_file = "results/regulon_summary.tsv"
os.makedirs(os.path.dirname(output_file), exist_ok=True)

with open(output_file, "w") as out:
    out.write("TF\tTotal genes\tActivados\tReprimidos\tTipo\tLista de genes\n")

    for TF in sorted(regulon):
        
        # Obtener datos
        genes = sorted(regulon[TF]["genes"])
        total = len(genes)
        activados = regulon[TF]["activados"]
        reprimidos = regulon[TF]["reprimidos"]

        # Determinar tipo de regulación
        if activados > 0 and reprimidos > 0:
            tipo = "dual"
        elif activados > 0:
            tipo = "activador"
        else:
            tipo = "represor"

        lista_genes = ", ".join(genes)
        out.write(f"{TF}\t{total}\t{activados}\t{reprimidos}\t{tipo}\t{lista_genes}\n")

Este código funciona.

El problema es que todas las tareas están mezcladas en un solo flujo:

  • lectura del archivo
  • validación
  • construcción de datos
  • generación de salida
  • clasificación del tipo de regulador

El problema del “código espagueti”

Cuando un programa crece sin una estructura clara:

  • todo ocurre en una sola secuencia
  • diferentes tareas quedan mezcladas
  • los límites entre partes del programa no son claros
  • modificar una parte puede afectar otras

A esto se le suele llamar informalmente código espagueti.

Pregunta guía

Antes de pensar en la sintaxis de funciones, hay que hacerse una pregunta más importante:

¿Qué tareas distintas está realizando este programa?

Ese cambio de pregunta es clave.

No empezamos por def.

Empezamos por identificar responsabilidades.

3. Cómo agrupar código: identificar responsabilidades

¿Qué es una responsabilidad?

Una responsabilidad es:

  • una tarea específica
  • con un propósito claro
  • que recibe ciertos datos
  • y produce un resultado o un efecto

En diseño de software, las funciones no se crean por sintaxis.

Las funciones se crean a partir de responsabilidades bien identificadas.

Criterios para agrupar código

Para decidir qué código debería quedar junto, usaremos estos criterios:

1. Cohesión

El código que realiza una misma tarea debería permanecer junto.

Ejemplo:

  • abrir archivo
  • recorrer líneas
  • limpiar texto
  • validar columnas
  • construir interactions

Todo eso pertenece al mismo bloque porque forma parte de una sola responsabilidad: leer y preparar interacciones.

2. Entradas y salidas claras

Un buen bloque tiene:

  • una entrada identificable
  • una salida identificable

Ejemplos:

  • archivo -> interactions
  • interactions -> regulon
  • regulon -> archivo de salida

3. Independencia

Un bloque bien definido debería poder entenderse sin tener que leer todo el programa al mismo tiempo.

Por ejemplo:

  • construir el regulon no debería depender de los detalles internos de lectura del archivo

4. Reutilización potencial

Si una parte podría usarse en otro contexto o al menos mejora mucho la legibilidad al separarse, conviene aislarla.

Ejemplo:

  • la lógica para determinar si un regulador es activador, represor o dual

5. Facilidad de prueba

Si una parte puede probarse por separado, normalmente está bien delimitada.

Esto significa que podemos ejecutar esa parte con ciertos datos y verificar si produce el resultado esperado, sin necesidad de correr todo el programa.

Por ejemplo:

  • podemos probar la función que construye el regulon usando una lista pequeña de interacciones
  • podemos probar la función que determina el tipo de regulador con distintos valores de entrada
  • podemos verificar que la salida generada tenga el formato correcto

Si tenemos una función:

def get_regulator_type(activados, reprimidos):
    ...

Podemos probarla directamente:

get_regulator_type(3, 0)  → debería ser "activador"  
get_regulator_type(0, 2)  → debería ser "represor"  
get_regulator_type(2, 2)  → debería ser "dual"

Sin necesidad de leer archivos o construir todo el regulon.

En desarrollo profesional, este tipo de pruebas se conocen como pruebas unitarias,
pero aquí lo importante es entender la idea:
probar piezas pequeñas de forma independiente.

4. Actividad guiada: detectar bloques en el script actual

Objetivo

Desarrollar criterio para pasar de leer código a entender su estructura lógica.

Paso 1: Identificar bloques

  1. Abre el archivo regulon_summary.py.
  2. Lee el script completo sin modificarlo todavía.
  3. Marca con comentarios los bloques que identifiques.

Ejemplo:

# leer interacciones

# construir regulon

# determinar tipo regulador

# generar salida

Preguntas guía

Para cada bloque responde:

  • ¿Qué hace esta parte del código?
  • ¿Qué datos recibe?
  • ¿Qué produce?
  • ¿De qué otra parte depende?

Discusión

No todos van a marcar exactamente los mismos bloques, y eso está bien.

El diseño requiere criterio.

Sin embargo, sí debería haber acuerdo en las responsabilidades principales.

Paso 2: Encontrar responsabilidades del programa

Una vez identificados los bloques, el siguiente paso es convertirlos en responsabilidades.

Un bloque es una parte del código. Una responsabilidad es una descripción clara de:

  • qué hace
  • qué datos usa
  • qué resultado produce

Ejemplo

Bloque identificado:

  • leer interacciones

Responsabilidad:

  • Leer un archivo, validar su contenido y construir una lista de interacciones.

Paso 3: Refinar responsabilidades

Una vez identificadas, conviene hacerlas más precisas usando una forma más estructurada:

  • nombre de la responsabilidad
  • entrada
  • salida

Ejemplo:

  • Responsabilidad: leer interacciones desde archivo
  • Entrada: archivo
  • Salida: lista de interactions

Dentro del código se vería como

# =========================================
# Responsabilidad: leer interacciones desde archivo
# Entrada: archivo
# Salida: lista de interactions
# =========================================

Esto ayuda a explicitar la intención del bloque antes de convertirlo en función.

Hagamoslo para todos los bloques.

Paso 4: Niveles de responsabilidad

No todas las responsabilidades tienen el mismo papel dentro del programa.

**Clasificacaión:

  • Alta

    • define el flujo principal del programa.
    • conecta entradas y salidas principales
    • suele corresponder a funciones principales

  • Media (subproceso importante)

    • forma parte de una responsabilidad mayor
    • puede o no convertirse en función
    • aquí empieza el criterio de diseño

  • Baja (detalle lógico)

    • lógica puntual
    • condiciones o cálculos específicos
    • puede ser función pequeña o inline

¿Cómo determinar el nivel?

Hazte esta pregunta:

¿Este bloque controla el flujo general, transforma datos o resuelve un detalle?

  • Flujo principal → Alta
  • Parte de un proceso → Media
  • Regla puntual → Baja

Preguntas:

Preguntas:

  • ¿Es el paso final del flujo?
  • ¿Conecta la estructura interna con el archivo de salida?

6. Concepto y sintaxis de funciones en Python

Una vez identificadas las responsabilidades, necesitamos la herramienta del lenguaje para implementarlas: las funciones.

¿Qué es una función?

Una función es un bloque de código con nombre que encapsula una tarea específica.

Sirve para:

  • organizar lógica
  • reutilizar código
  • separar responsabilidades
  • hacer más claro el programa

Sintaxis básica

def nombre_funcion(parametros):
    # cuerpo de la función
    return resultado

Elementos importantes

palabra reservada def

La palabra clave def indica que se está definiendo una función.

Nombre de la función

Debe describir claramente qué hace.

Buenos ejemplos en este problema:

  • load_interactions
  • build_regulon
  • write_summary

Parámetros

Son los datos de entrada que la función necesita para trabajar.

Ejemplo:

def load_interactions(input_file):

Aquí input_file es el parámetro.

Cuerpo de la función

Es el bloque de instrucciones que realiza la tarea.

palabra reservada return

Se usa para devolver un resultado.

Ejemplo:

return interactions

Modelo mental: una función transforma datos

En este programa, podemos pensar las funciones así:

archivo -> interactions -> regulon -> archivo de salida

Cada función toma algo, lo transforma y produce un resultado.

Ejemplos:

  • load_interactions(input_file) -> interactions
  • build_regulon(interactions) -> regulon
  • get_regulator_type(activados, reprimidos) -> tipo

Este modelo mental es muy útil en bioinformática y ciencia de datos.

Funciones con y sin return

Funciones que devuelven un valor

Ejemplo:

def get_regulator_type(activados, reprimidos):
    if activados > 0 and reprimidos > 0:
        return "dual"
    elif activados > 0:
        return "activador"
    else:
        return "represor"

Aquí la función devuelve una clasificación.

Funciones que producen un efecto

Ejemplo:

def write_summary(regulon, output_file):
    ...

En este caso, la función no necesita regresar un dato nuevo. Su trabajo principal es escribir un archivo.

Variables locales

Las variables creadas dentro de una función existen solo dentro de esa función.

Esto ayuda a:

  • evitar efectos inesperados
  • reducir dependencias innecesarias
  • hacer el código más seguro y claro

Regla práctica

Una función debe hacer una sola cosa y hacerla bien.

Eso no significa que deba tener una sola línea.

Significa que debe tener una responsabilidad clara.

7. 📝 Documentación de funciones: docstrings

Una vez que empezamos a organizar el código en funciones, surge una nueva necesidad:

explicar claramente qué hace cada función

Para esto se utilizan los docstrings.

¿Qué es un docstring?

Un docstring es un texto que se escribe al inicio de una función para describir:

  • qué hace
  • qué recibe
  • qué devuelve

Se define usando comillas triples:

""" ... """

Ejemplo en nuestro contexto

def load_interactions(input_file):
    """Lee un archivo TSV, valida su contenido y devuelve una lista de interacciones."""

¿Por qué son importantes?

Los docstrings ayudan a:

  • entender el código más rápido
  • facilitar mantenimiento
  • comunicar intención
  • mejorar el uso de herramientas de IA
  • generar documentación automática

¿Qué debería incluir un buen docstring?

Como mínimo:

  • qué hace la función
  • qué parámetros recibe
  • qué devuelve

Formatos comunes de docstrings

Existen varios estilos. No es obligatorio usar uno complejo al inicio, pero es importante conocerlos.

1. Formato simple (recomendado)

def build_regulon(interactions): """ Construye un diccionario de regulones a partir de una lista de interacciones.

Parámetros:
interactions: lista de tuplas (TF, gene, effect)

Retorna:
dict: estructura del regulon
"""

✔ claro
✔ fácil de leer
✔ suficiente para empezar

2. Formato Google

def getregulatortype(activados, reprimidos): """ Determina el tipo de regulador.

Args:
    activados (int): número de genes activados
    reprimidos (int): número de genes reprimidos

Returns:
    str: tipo de regulador (activador, represor o dual)

✔ muy usado en la industria
✔ compatible con muchas herramientas

3. Formato NumPy

def writesummary(regulon, outputfile): """ Escribe el resumen del regulon en un archivo.

Parameters
----------
regulon : dict
    Estructura del regulon
output_file : str
    Ruta del archivo de salida

Returns

✔ común en ciencia de datos
✔ más estructurado

Para este curso conviene elegir uno solo para mantener consistencia.

Buenas prácticas

  • usar frases claras y cortas
  • describir la responsabilidad de la función
  • no repetir el nombre de la función en la descripción
  • mantener consistencia en todo el proyecto

Conexión con el diseño

si no puedes escribir claramente qué hace una función en su docstring,
probablemente la función no está bien definida

9. Refactorización incremental sobre el código real

Regla general

Nunca conviene refactorizar todo de golpe.

La estrategia correcta es avanzar con cambios pequeños y verificables.

Paso 0. Leer antes de mover

Antes de escribir una sola función:

  • leer el script completo
  • identificar bloques
  • entender dependencias

Paso 1. Extraer load_interactions()

Conviene comenzar con la lectura y validación porque:

  • tiene una entrada clara: input_file
  • tiene una salida clara: interactions
  • agrupa tareas relacionadas

Qué debe hacer

  • verificar existencia del archivo
  • abrir el archivo
  • recorrer líneas
  • limpiar texto
  • ignorar comentarios y encabezado
  • validar columnas
  • validar effect
  • construir la lista interactions

Resultado esperado


def load_interactions(input_file):
    """Carga las interacciones del archivo TSV de reguladores a genes.

    Lee `input_file`, ignora comentarios (#), encabezado y filas inválidas.
    Valida que cada fila tenga al menos 7 columnas y efecto sea '+' o '-'.

    Args:
        input_file (str): Ruta al archivo TSV de entrada.

    Returns:
        list[tuple[str, str, str]]: Lista de tuplas (TF, gene, effect).

    Raises:
        SystemExit: Si el archivo no existe.
    """
    interactions = []

    if not os.path.exists(input_file):
        print("Error: archivo no encontrado")
        exit(1)
    else:
        with open(input_file) as f:
            for line in f:
                line = line.strip()

                # Ignorar líneas vacías
                if not line:
                    continue

                # Ignorar comentarios
                if line.startswith("#"):
                    continue

                # Ignorar encabezado
                if line.startswith("1)regulatorId"):
                    continue

                fields = line.split("\t")

                # Validar número mínimo de columnas
                if len(fields) <= 6:
                    continue

                TF = fields[1]
                gene = fields[4]
                effect = fields[5]

                # Validar effect
                if effect not in ["+", "-"]:
                    continue

                interactions.append((TF, gene, effect))

     return interactions

Luego, en el flujo principal:

interactions = load_interactions(input_file)

Después de este cambio, hay que probar que todo siga funcionando igual.

Paso 2. Extraer build_regulon()

Después, se separa la transformación de interactions a regulon.

Qué debe hacer

  • inicializar estructura por TF
  • agregar genes
  • contar activados y reprimidos

Idea clave

Esta función no debería leer ni escribir archivos.

Su responsabilidad es solo transformar datos.

Genera la función

Estructura esperada

def build_regulon(interactions):
    """Construye la estructura regulon a partir de las interacciones.

    Args:
        interactions (list[tuple[str, str, str]]): Lista de interacciones.

    Returns:
        dict: Estructura del regulon.
    """
    regulon = {}

    for TF, gene, effect in interactions:
        if TF not in regulon:
            regulon[TF] = {
                "genes": [],
                "activados": 0,
                "reprimidos": 0,
            }

        regulon[TF]["genes"].append(gene)

        if effect == "+":
            regulon[TF]["activados"] += 1
        elif effect == "-":
            regulon[TF]["reprimidos"] += 1

    return regulon

Separarla puede:

  • mejorar legibilidad
  • facilitar pruebas
  • hacer más clara la intención del código

Pregunta central

Contexto del flujo actual

Preguntas para discutir

  1. ¿Quién necesita realmente el valor de tipo?

    • ¿se necesita para construir el regulon?
    • ¿o solo para escribir la salida?

  2. ¿Qué representa tipo dentro del programa?

    • ¿es parte de los datos internos del modelo (regulon)?
    • ¿o es un valor derivado que solo se usa para mostrarlo en el reporte?

  3. Observa estas dos posibilidades de diseño:

    Opción A:

    main() -> load_interactions() -> build_regulon() -> get_regulator_type() -> write_summary()

    Opción B:

    main() -> load_interactions() -> build_regulon() -> write_summary() -> get_regulator_type()

  4. Si get_regulator_type() se usa dentro de build_regulon():

    • el regulon queda más completo desde antes
    • write_summary() solo escribe datos ya preparados

  5. Si get_regulator_type() se usa dentro de write_summary():

    • el regulon se mantiene más simple
    • write_summary() calcula un valor derivado para el reporte final

Criterio de decisión

No te fijes solo en qué datos usa la función.

Piensa sobre todo en esto:

  • ¿qué responsabilidad está ayudando a cumplir?
  • ¿está construyendo datos internos?
  • ¿o está ayudando a generar la salida?

La ubicación de una función no depende únicamente de los datos que recibe, sino de la responsabilidad a la que sirve dentro del programa.

Paso 4. Extraer write_summary()

Finalmente, se separa la escritura del archivo de salida.

Qué debe hacer

  • crear el directorio de salida si hace falta
  • abrir el archivo
  • escribir encabezado
  • recorrer el regulon
  • calcular los datos finales por TF
  • llamar a getregulatortype
  • escribir cada línea

Paso 5. Crear main()

Una vez separadas las funciones principales, main() coordina el flujo completo.

Antes y después: cambio estructural esperado

Antes

script lineal
 ├── lectura
 ├── validación
 ├── construcción
 ├── clasificación
 └── salida

Después

main()
 ├── load_interactions(input_file)
 ├── build_regulon(interactions)
 ├── write_summary(regulon, output_file)
 └──     └── get_regulator_type(activados, reprimidos)   [opcional]

La funcionalidad es la misma.

Lo que cambia es la organización.

10. Uso de TODOs como puente entre análisis e implementación

Los TODOs pueden ayudar a traducir el análisis de responsabilidades a pasos concretos de trabajo.

En refactorización usamos verbos como:

  • extraer
  • mover
  • reorganizar
  • separar
  • encapsular

En implementación usamos:

  • crear
  • implementar
  • desarrollar

¿Por qué importa el verbo que usamos?

El verbo comunica la intención del cambio.

Esto es importante tanto para:

  • el desarrollador
  • como para herramientas de IA (como Copilot)

Ejemplo:

# TODO: extraer lectura y validación del archivo a load_interactions()
# TODO: extraer construcción del regulon a build_regulon()
# TODO: evaluar si conviene crear get_regulator_type()
# TODO: extraer escritura del archivo a write_summary()
# TODO: crear main() para coordinar el flujo

Aquí los TODOs no representan “código faltante” desde cero.

Representan decisiones de refactorización pendientes.

Los TODOs no solo sirven como recordatorio.

También pueden usarse para guiar herramientas de IA como Copilot.

En la siguiente sección veremos cómo utilizarlos correctamente.

12. Validación: pruebas de regresión

La condición final de toda esta lección es muy simple:

con la misma entrada, el programa debe producir la misma salida

Por eso, después de cada cambio importante, hay que verificar:

  • ¿el archivo de salida sigue teniendo el mismo formato?
  • ¿los conteos siguen siendo correctos?
  • ¿la clasificación activador / represor / dual se mantiene?
  • ¿los casos de prueba anteriores siguen pasando?

Eso confirma que hubo refactorización real y no un cambio accidental de comportamiento.

13. Uso de IA con criterio

Estrategia correcta

  1. analizar el problema
  2. identificar responsabilidades
  3. diseñar funciones
  4. pedir ayuda parcial a la IA
  5. revisar e integrar manualmente
  6. validar con pruebas

Riesgo común al usar IA

La IA puede:

  • cambiar lógica sin avisar
  • mezclar responsabilidades
  • proponer nombres confusos
  • alterar la salida del programa

Por eso:

La IA propone; tú decides y verificas

Buen prompt

Tengo un programa en Python que ya funciona.

No quiero cambiar su funcionalidad.
Quiero refactorizarlo para organizarlo en funciones.

Estas son las responsabilidades identificadas:
- leer interacciones desde archivo
- construir regulon
- determinar tipo de regulador
- escribir salida

Propón funciones con:
- nombre
- parámetros
- valor de retorno

No escribas el programa completo.

Mal prompt

refactoriza este código

Checklist al usar IA

  • ¿cambió la lógica?
  • ¿cambió la salida?
  • ¿respeta el diseño?
  • ¿cada función tiene una responsabilidad clara?

O bien, si implementas usando los TODOs, una vez ya integrados en el codigo podrias dar el siguiente prompt: 

Extrae únicamente el bloque de lectura y validación del archivo a una función llamada load_interactions(input_file). No cambies la lógica.

14. Conclusión

Antes de esta lección, el objetivo era escribir un programa que funcionara.

Ahora damos un paso más:

aprender a organizar un programa real para que siga siendo comprensible y mantenible cuando crece

La idea central es esta:

  1. primero entiendes qué hace el programa
  2. luego identificas responsabilidades
  3. después diseñas funciones
  4. finalmente refactorizas sin cambiar el comportamiento

Ese es el paso de escribir código que funciona a empezar a construir software de mejor calidad.
