Actividad — Evolución de un programa con Git

Objetivo de la actividad

En esta actividad vamos a aprender a desarrollar un programa de manera incremental utilizando Git para registrar la evolución del código. En lugar de crear múltiples versiones del programa en archivos distintos, trabajaremos siempre sobre un mismo archivo que irá mejorando mediante commits sucesivos.

Además, en este curso utilizamos uv como infraestructura estándar para:

  • crear proyectos de Python
  • gestionar el entorno
  • ejecutar programas dentro del proyecto

Relación con las buenas prácticas del curso

En la sesión Introducción al desarrollo de software, se explicó que desarrollar software no consiste solo en escribir código. Un proyecto científico debe incluir al menos tres elementos fundamentales:

  1. Documentación del proyecto (README.md)
  2. Documento de contexto o análisis del problema (contexto.md)
  3. Pruebas o casos de prueba que permitan verificar el comportamiento del programa

Estos elementos forman parte del ciclo de vida del software, particularmente de las etapas de requisitos, análisis/diseño y pruebas. fileciteturn0file0

En esta actividad comenzaremos a practicar estas buenas prácticas en una forma sencilla.

El proyecto incluirá:

  • README.md — descripción general del proyecto
  • contexto.md — definición del problema, datos de entrada y requisitos
  • kmer_lab.py — implementación del programa

A lo largo de la actividad, cada nueva funcionalidad del programa deberá reflejarse también en el archivo de contexto, para mantener claro qué problema se está resolviendo y cómo evoluciona la solución.

Dinámica de trabajo

Durante esta actividad el programa evolucionará en tres etapas.

Cada etapa implicará:

  • actualizar el archivo de contexto
  • modificar el programa
  • registrar los cambios mediante un commit

El objetivo es que en GitHub se pueda observar cómo el programa evoluciona paso a paso.

Etapa 1 — Preparación del proyecto

Primero prepararemos la infraestructura del proyecto.

Crear el proyecto con uv

En la terminal:

uv init kmer_lab
cd kmer_lab

Esto crea la estructura básica del proyecto.

Crear los archivos de la actividad

touch contexto.md kmer_lab.py

La estructura mínima del proyecto debe quedar así:

kmer_lab/
  pyproject.toml
  README.md
  contexto.md
  kmer_lab.py

Inicializar el repositorio

Si Git no fue inicializado automáticamente:

git init

Primer commit

Este commit registra la estructura inicial del proyecto.

git add .
git commit -m "init: project structure"

Conectar el proyecto con GitHub

Ahora conectaremos el repositorio local con GitHub. Hay dos formas válidas de hacerlo en esta actividad.

Opción A — Desde la línea de comando

  1. Crear un repositorio vacío en GitHub llamado kmer_lab.
  2. Copiar la URL del repositorio.
  3. Ejecutar en terminal:
git branch -M main
git remote add origin <URL_DEL_REPOSITORIO>
git push -u origin main

Opción B — Desde Visual Studio Code

También se puede conectar el repositorio usando la interfaz de Visual Studio Code, por ejemplo desde la sección de control de versiones o con las opciones para publicar el repositorio en GitHub.

Lo importante es que al final:

  • el repositorio local quede conectado con GitHub
  • el commit inicial aparezca en el repositorio remoto

Etapa 2 — Generar k-mers

En esta etapa vamos a construir la primera versión funcional del programa.

El objetivo es escribir un programa que:

  • lea una secuencia de DNA
  • lea el valor de k
  • genere todos los k-mers posibles de longitud k
  • muestre cada k-mer en pantalla

A) Crear el documento de contexto (antes de programar)

Antes de escribir código, se debe crear y completar el archivo:

contexto.md

Este documento corresponde a las etapas de requisitos, análisis y diseño del ciclo de vida del software.

Su objetivo es definir claramente el problema antes de comenzar a programar.

El archivo contexto.md debe incluir las siguientes secciones.

1. Problema

Descripción breve del problema que se quiere resolver.

Ejemplo:

Generar todos los fragmentos consecutivos de longitud k (k-mers) a partir de una secuencia de DNA proporcionada por el usuario.

2. Datos de entrada

Especificar qué información recibe el programa.

Para esta actividad:

  • una secuencia de DNA introducida por el usuario
  • un número entero k que representa el tamaño del k-mer

3. Requisitos funcionales

Los requisitos funcionales describen qué debe hacer el programa.

Ejemplo para esta primera versión:

  • leer una secuencia de DNA
  • leer el valor de k
  • recorrer la secuencia
  • generar todos los k-mers de longitud k
  • mostrar cada k-mer en pantalla

4. Requisitos no funcionales

Los requisitos no funcionales describen cómo debe comportarse el programa o restricciones que debe cumplir.

Ejemplos:

  • el programa debe aceptar secuencias en mayúsculas o minúsculas
  • la secuencia debe convertirse a mayúsculas
  • el código debe ser legible y seguir convenciones básicas de estilo

5. Algoritmo

Describir en lenguaje simple los pasos de la solución.

Ejemplo:

  1. Leer la secuencia de DNA.
  2. Leer el valor de k.
  3. Recorrer la secuencia desde la posición 0 hasta len(seq) - k.
  4. En cada posición, extraer un fragmento de longitud k.
  5. Mostrar el k-mer generado.

B) Casos de prueba

Antes de programar se deben definir casos de prueba para verificar que el programa funciona correctamente.

Para esto genera un documento de pruebas.md

Incluir al menos:

Caso normal

Entrada
DNA: ATGCG
k: 3

Salida esperada
ATG
TGC
GCG

Caso límite

  • Cuando k es igual a la longitud de la secuencia.
DNA: ATG
k: 3

Salida esperada
ATG
  • Agrega otros casos limites

Caso de trazado manual

Realizar un trazado manual del algoritmo.

Ejemplo:

seq = ATGCG
k = 3

Posición 0 → ATG
Posición 1 → TGC
Posición 2 → GCG

Este ejercicio ayuda a verificar que el algoritmo es correcto antes de programar.

Etapa 3 — Extensión: mejor k-mer por contenido GC

En esta etapa el programa se extenderá para identificar cuál de los k-mers generados tiene el mayor contenido de G y C.

Ampliación del contexto

Se debe actualizar contexto.md para reflejar esta nueva versión.

Problema ampliado

Además de generar todos los k-mers, ahora queremos evaluar cada uno para determinar cuál tiene mayor contenido GC.

Datos de entrada

  • una secuencia de DNA proporcionada por el usuario
  • un valor entero k

Nuevos requisitos

A los requisitos anteriores se agregan los siguientes:

  • calcular cuántos nucleótidos G o C tiene cada k-mer
  • comparar el valor GC de cada k-mer con el mejor encontrado hasta el momento
  • guardar el mejor k-mer
  • guardar la posición donde aparece
  • mostrar al final el mejor resultado encontrado

Salida Esperada:

kmer=ATG GC=1
Best so far → ATG (GC=1)

kmer=TGC GC=2
Best so far → TGC (GC=2)

kmer=GCG GC=3
Best so far → GCG (GC=3)

Final best k-mer: GCG
Position: 2
GC count: 3

Algoritmo esperado

  1. Leer la secuencia de DNA.
  2. Leer el valor de k.
  3. Preparar variables para guardar el mejor resultado encontrado hasta el momento.
  4. Recorrer la secuencia usando una ventana de tamaño k.
  5. Obtener el k-mer de la posición actual.
  6. Contar cuántas letras G o C tiene ese k-mer.
  7. Mostrar el k-mer actual junto con su valor GC.
  8. Comparar ese valor con el mejor valor guardado.
  9. Si el k-mer actual tiene mayor contenido GC, actualizar el mejor resultado.
  10. Mostrar el mensaje Best so far con el mejor k-mer registrado hasta ese momento.
  11. Al final, mostrar el mejor k-mer, su posición y su contenido GC.

Trabajo del alumno

En esta etapa el estudiante deberá:

  • actualizar contexto.md con los nuevos requisitos
  • extender kmer_lab.py
  • probar el programa
  • registrar los cambios en Git
uv run python kmer_lab.py
git add kmer_lab.py contexto.md
git commit -m "feat: compute best GC k-mer"
git push

Etapa 4 — Extensión final: visualización de ventana deslizante

En esta etapa el programa se extenderá para mostrar de manera más explícita cómo se mueve la ventana sobre la secuencia y cómo cambia el mejor resultado encontrado.

Se debe volver a actualizar contexto.md.

Problema ampliado
