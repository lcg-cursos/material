🧠 3. Contexto conceptual: tipos de errores en Python

Antes de hablar de try y except, conviene detenernos en una idea importante:

No todos los errores son del mismo tipo, y no todos se resuelven de la misma manera.

En programación, distinguir el tipo de error nos ayuda a decidir mejor:

  • qué revisar primero,
  • qué corregir en el código,
  • qué informar al usuario,
  • y cuándo conviene detener el programa o continuar.

En esta lección trabajaremos con tres categorías muy útiles para analizar programas reales.

3.1 Errores de sintaxis

Son errores en la forma en que está escrito el código.

Python no puede interpretar el programa porque la instrucción está mal construida. En este caso, el programa ni siquiera comienza a ejecutarse.

Ejemplo

if not os.path.exists(input_file)
    print("El archivo no existe")

¿Qué se intenta hacer aquí?

Se quiere validar si el archivo de entrada existe. Si no existe, se desea imprimir un mensaje para el usuario.

¿Qué problema hay?

Falta : al final de la línea del if.

¿Qué ocurre?

Python detecta el problema antes de ejecutar el programa y lanza un error de sintaxis.

Idea clave

Un error de sintaxis no se maneja con try / except.

Primero hay que corregir el código fuente.

3.2 Errores lógicos

Son errores donde el programa sí corre, pero hace algo incorrecto o distinto de lo esperado.

Estos errores son especialmente peligrosos porque a veces no producen traceback (ruta del error) ni mensajes visibles. El programa parece funcionar, pero el resultado está mal.

Ejemplo

if effect == "+":
    regulon[TF]["reprimidos"] += 1

¿Qué se intenta hacer aquí?

Se quiere contar cuántos genes son activados por un factor de transcripción.

¿Qué problema hay?

La condición revisa correctamente effect == "+", pero se incrementa el contador equivocado: reprimidos en vez de activados.

¿Qué ocurre?

El programa sigue corriendo y produce salida, pero los resultados biológicos serían incorrectos.

Idea clave

Los errores lógicos no se arreglan con excepciones.

Se detectan mediante:

  • revisión del código,
  • CASOS DE PRUEBA,
  • comparación con RESULTADOS ESPERADOS,
  • inspección manual de salidas.

3.3 Errores en tiempo de ejecución

Son errores que ocurren mientras el programa ya está corriendo.

Aquí el código puede estar bien escrito, pero el entorno o los datos generan una situación que impide continuar normalmente.

Estos sí son los errores que suelen manejarse con excepciones (try/except).

Ejemplo 1

with open(input_file) as f:
    for line in f:
        print(line)

¿Qué se intenta hacer aquí?

Se quiere abrir el archivo de entrada y recorrerlo línea por línea.

¿Qué problema podría ocurrir?

Si el archivo no existe, si la ruta es incorrecta o si no hay permisos para leerlo, open() puede fallar.

¿Qué ocurre?

Python lanza una excepción en tiempo de ejecución, por ejemplo FileNotFoundError u otra relacionada con el sistema de archivos.

Traceback (most recent call last):
  File "script.py", line 1, in <module>
    with open(input_file) as f:
FileNotFoundError: [Errno 2] No such file or directory: 'archivo.txt'

🧭 Cómo leer un traceback (de abajo hacia arriba)

  1. 🔴 Última línea → identifica el error

    • Tipo de error (ej. FileNotFoundError)
    • Mensaje (qué salió mal)

  2. 🟡 Línea(s) intermedia(s) → ubica dónde ocurrió

    • La instrucción exacta que causó el error
    • Función o bloque donde pasó

  3. 🔵 Primeras líneas → entiende el contexto

    • Archivo (script.py)
    • Número de línea
    • Flujo de ejecución (cómo llegó ahí)

🎯 Regla rápida

  • Abajo = qué pasó
  • En medio = dónde pasó
  • Arriba = cómo llegó ahí

Idea clave

Aquí sí tiene sentido usar try / except, porque el problema no siempre depende del código, sino del contexto real en que se ejecuta.

Ejemplo 2

value = int(user_text)

¿Qué se intenta hacer aquí?

Convertir un texto a entero.

¿Qué problema podría ocurrir?

Si user_text contiene letras o símbolos no numéricos, la conversión falla.

¿Qué ocurre?

Python lanza una excepción ValueError.

Traceback (most recent call last):
  File "script.py", line 3, in <module>
    value = int(user_text)
ValueError: invalid literal for int() with base 10: 'abc'

🧭 Cómo leer el traceback para: value = int(user_text)

  1. 🔴 Última línea → identifica el error

    • Tipo de error: ValueError
    • Mensaje típico: ValueError: invalid literal for int() with base 10: 'abc'
    • 👉 Significa que el texto no se puede convertir a entero

  2. 🟡 Línea(s) intermedia(s) → ubica dónde ocurrió

    • Línea problemática: python value = int(user_text)
    • 👉 Aquí falló la conversión

  3. 🔵 Primeras líneas → entiende el contexto

    • Archivo y número de línea donde ocurrió
    • Flujo del programa hasta llegar a esa conversión

🎯 Regla rápida

  • Abajo = qué pasó → no se pudo convertir a entero
  • En medio = dónde pasó → en int(user_text)
  • Arriba = cómo llegó ahí → flujo del programa

⚠️ Causa común

  • user_text = "abc"
  • user_text = "12.5"
  • user_text = ""

👉 Todo lo que no sea un entero válido produce este error

Idea clave

No todo error de ejecución tiene que ver con archivos. También pueden surgir al procesar datos, convertir tipos o usar argumentos inválidos.

🧠 Buenas prácticas para minimizar errores de ejecución

1. ✅ Validar antes de operar

  • Verifica condiciones antes de ejecutar operaciones críticas
  • Ejemplo: listas vacías, tipos de datos, valores válidos

👉 Evita errores predecibles

✔ Correcto:

if len(valores) == 0:
    return None

2. 🧩 No uses try/except como sustituto de lógica

  • No captures errores que podrías prevenir
  • Evita usar excepciones para controlar flujo normal

❌ Mala práctica:

try:
    promedio = sum(valores) / len(valores)
except:
    promedio = 0

✔ Correcto:

if len(valores) == 0:
    promedio = 0
else:
    promedio = sum(valores) / len(valores)

3. 🎯 Captura errores específicos

  • No uses except: genérico
  • Captura solo lo que esperas

❌ Mala práctica:

try:
    archivo = open("datos.txt")
except:
    print("Error")

✔ Correcto:

try:
    archivo = open("datos.txt")
except FileNotFoundError:
    print("El archivo no existe")

4. 🗣️ Da mensajes claros al usuario

  • Explica qué salió mal y qué hacer
  • Evita mensajes crípticos

❌ Mala práctica:

print("Error")

✔ Correcto:

print("Error: la lista está vacía, no se puede calcular el promedio")

5. 📂 Asume que el entorno puede fallar

  • Archivos pueden no existir
  • Permisos pueden fallar
  • Datos pueden estar corruptos

👉 Aquí sí usa try/except

✔ Correcto:

try:
    with open("datos.txt") as f:
        contenido = f.read()
except FileNotFoundError:
    print("Archivo no encontrado")

6. 🔍 Valida entradas del usuario

  • Nunca confíes en input externo
  • Convierte y valida antes de usar

❌ Mala práctica:

edad = int(input())

✔ Correcto:

texto = input()
if texto.isdigit():
    edad = int(texto)
else:
    print("Entrada inválida")

7. 🧪 Usa casos de prueba simples

  • Prueba con:
    • valores vacíos
    • valores extremos
    • datos incorrectos

👉 Esto ayuda a detectar errores lógicos y de ejecución

✔ Correcto:

assert calcular_promedio([1,2,3]) == 2
assert calcular_promedio([]) is None

8. 🧱 Diseña fallos controlados

  • Decide qué debe hacer el programa si algo falla:
    • detenerse
    • continuar
    • usar valores por defecto

✔ Correcto:

if not datos:
    print("No hay datos disponibles")
    return

🎯 Regla de oro

Errores predecibles → validación
Errores inevitables → manejo con excepciones

🛠️ 6. Estrategias de manejo

No todos los errores se atienden igual. En programación real, suele ser útil combinar dos estrategias:

  • validación previa, cuando podemos revisar una condición antes de ejecutar una operación riesgosa,
  • manejo de excepciones, cuando el error solo se conoce al intentar ejecutar la operación.

La idea no es elegir una u otra como si fueran excluyentes, sino entender en qué casos conviene cada una.

6.1 Validación previa

La validación previa consiste en revisar una condición antes de continuar.

Ejemplo

if not os.path.exists(input_file):
    print("Error: archivo no existe")
    exit(1)

¿Qué se intenta hacer aquí?

Antes de leer el archivo, se verifica si la ruta existe.

¿Por qué es útil?

Porque permite dar un mensaje claro al usuario desde el inicio, sin dejar que el programa avance innecesariamente.

Limitación importante

Aunque esta validación ayuda, no garantiza por completo que el archivo podrá abrirse. Por ejemplo, el archivo puede existir pero no tener permisos de lectura.

Idea clave

La validación previa mejora la experiencia del usuario, pero no reemplaza completamente el manejo de excepciones.

6.2 Manejo de excepciones

El manejo de excepciones permite controlar errores que aparecen al ejecutar una operación real.

Ejemplo

try:
    with open(filename) as f:
        for line in f:
            print(line)
except FileNotFoundError:
    print("Error: el archivo no fue encontrado")

¿Qué se intenta hacer aquí?

Abrir un archivo y recorrer su contenido línea por línea.

¿Qué hace try?

Indica el bloque donde podría ocurrir un error.

¿Qué hace except FileNotFoundError?

Captura específicamente el caso en que el archivo no existe.

¿Por qué esto es mejor que dejar el traceback crudo?

Porque permite mostrar un mensaje más claro y controlado para el usuario.

6.3 ¿Cuándo conviene usar validación y cuándo excepción?

Una forma práctica de decidirlo es esta:

  • Valida antes cuando la condición es fácil, directa y mejora el mensaje al usuario.
  • Usa excepción cuando la operación depende del sistema, del entorno o de datos que solo se conocen al ejecutarse.

Ejemplo en este programa

  • Revisar si la ruta existe → validación previa.
  • Intentar abrir el archivo → manejo de excepción.
  • Intentar escribir la salida → manejo de excepción.

6.4 Regla importante

No todo se valida antes, ni todo se captura después.

La robustez de un programa suele surgir de combinar ambas estrategias con criterio.

💻 7. Implementación guiada (live coding)

En esta parte no vamos a reescribir todo el programa de golpe.

La idea es mejorar el manejo de errores de forma gradual, entendiendo por qué cada cambio tiene sentido.

Paso 1: manejar errores al leer el archivo de entrada

def load_interactions(filename):
    interactions = []

    try:
        with open(filename) as f:
            for line in f:
                line = line.strip()

                if not line:
                    continue

                if line.startswith("#"):
                    continue

                if line.startswith("1)regulatorId"):
                    continue

                fields = line.split("   ")

                if len(fields) <= 6:
                    continue

                TF = fields[1]
                gene = fields[4]
                effect = fields[5]

                if effect not in ["+", "-", "+-"]:
                    continue

                interactions.append((TF, gene, effect))

    except FileNotFoundError:
        raise RuntimeError(f"No se encontró el archivo de entrada: {filename}")
    except PermissionError:
        raise RuntimeError(f"No hay permisos para leer el archivo: {filename}")
    except OSError as e:
        raise RuntimeError(f"Ocurrió un error al leer el archivo {filename}: {e}")

    return interactions

¿Qué se está haciendo aquí?

Se mantiene la lógica original de lectura, pero ahora la apertura del archivo queda protegida por un bloque try.

¿Qué mejora aporta?

Si algo falla al abrir o leer el archivo, el programa puede comunicarlo con un mensaje más comprensible que un traceback técnico sin contexto.

¿Por qué usar RuntimeError aquí?

Porque permite relanzar un error con un mensaje más claro para el flujo general del programa. La función detecta el problema y main() puede decidir cómo reaccionar.

Paso 2: manejar errores al escribir la salida

def write_summary(regulon, output_file):
    try:
        os.makedirs(os.path.dirname(output_file), exist_ok=True)

        with open(output_file, "w") as out:
            out.write("TF   Total genes Activados   Reprimidos  Tipo    Lista de genes
")

            for TF in sorted(regulon):
                genes = sorted(regulon[TF]["genes"])
                total = len(genes)
                activados = regulon[TF]["activados"]
                reprimidos = regulon[TF]["reprimidos"]

                if activados > 0 and reprimidos > 0:
                    tipo = "dual"
                elif activados > 0:
                    tipo = "activador"
                else:
                    tipo = "represor"

                lista_genes = ", ".join(genes)
                out.write(
                    f"{TF}  {total} {activados} {reprimidos}    {tipo}  {lista_genes}
"
                )

    except PermissionError:
        raise RuntimeError(f"No hay permisos para escribir el archivo: {output_file}")
    except OSError as e:
        raise RuntimeError(f"Ocurrió un error al escribir el archivo {output_file}: {e}")

¿Qué se está haciendo aquí?

Se protege tanto la creación del directorio como la escritura del archivo de salida.

¿Por qué esto es importante?

Porque un programa no solo debe manejar bien la entrada. También debe responder con claridad cuando falla al guardar resultados.

Observación útil para clase

  • ¿qué pasa si la carpeta no existe?,
  • ¿qué pasa si sí existe, pero no tenemos permisos?,

Paso 3: centralizar el manejo en main()

def main():
    args = parse_arguments()

    input_file = args.input_file
    output_file = args.output_file
    min_genes = args.min_genes

    if not os.path.exists(input_file):
        print(f"Error: el archivo no existe -> {input_file}")
        exit(1)

    try:
        interactions = load_interactions(input_file)
        regulon = build_regulon(interactions)

        nuevo_regulon = {}

        for TF, data in regulon.items():
            num_genes = len(data["genes"])
            if num_genes >= min_genes:
                nuevo_regulon[TF] = data

        regulon = nuevo_regulon

        write_summary(regulon, output_file)

    except RuntimeError as e:
        print(f"Error: {e}")
        exit(1)

¿Qué se está haciendo aquí?

main() coordina el flujo principal del programa y decide qué hacer cuando una función reporta un problema.

¿Por qué es buena idea centralizar aquí?

Porque evita que cada función imprima por su cuenta mensajes desordenados. Así, las funciones pueden enfocarse en su responsabilidad y main() en la interacción final con el usuario.

Idea importante para discutir

Aquí vale la pena preguntar:

  • ¿conviene que las funciones impriman errores?
  • ¿o conviene que lancen excepciones y que main() decida?

Esa discusión ayuda a entender separación de responsabilidades.

Paso 4: pensar en advertencias, no solo en fallos fatales

No todo problema debe terminar el programa.

Por ejemplo, si después del filtro --min_genes no queda ningún regulador, eso no necesariamente es una excepción. Puede ser simplemente un resultado vacío que conviene comunicar mejor.

Ejemplo

if not regulon:
    print("Advertencia: no hay reguladores que cumplan el filtro indicado.")

¿Qué se intenta hacer aquí?

Avisar al usuario que el programa sí corrió, pero que no produjo resultados útiles bajo las condiciones dadas.

¿Por qué esto es valioso?

Porque ayuda a distinguir entre:

  • un programa que falló,
  • y un programa que terminó bien, pero no encontró datos relevantes.

🧪 8. Casos de prueba

En esta lección, los casos de prueba no solo sirven para revisar si el programa funciona en condiciones normales. También sirven para verificar si el programa responde bien cuando algo sale mal.

La pregunta ya no es solo:

¿Corre el programa?

Sino también:

¿Falla de manera clara, controlada y útil?

Caso 1: el archivo no existe

Qué se va a hacer

Ejecutar el programa con una ruta de entrada que no corresponde a ningún archivo real.

Ejemplo

python src/regulon_summary.py data/inexistente.tsv results/output.tsv

Qué se espera

  • El programa no debe continuar.
  • Debe mostrar un mensaje claro.
  • Debe terminar con error controlado.

Qué se discute aquí

Este caso permite distinguir entre validación previa y excepción. También ayuda a discutir si el mensaje es suficientemente claro para el usuario.

Caso 2: el archivo existe, pero contiene filas problemáticas

Qué se va a hacer

Probar un archivo que tenga:

  • líneas vacías,
  • comentarios,
  • filas con pocas columnas,
  • efectos inválidos.

Qué se espera

  • El programa no debe romperse por una sola línea mala.
  • Debe seguir procesando el resto.
  • Se puede discutir si conviene advertir cuántas líneas fueron ignoradas.

Qué se discute aquí

Este caso es ideal para hablar sobre decisiones de diseño:

  • ¿ignorar silenciosamente?
  • ¿contar líneas inválidas?
  • ¿detener el programa en modo estricto?

Caso 3: la ruta de salida no se puede escribir

Qué se va a hacer

Intentar guardar el archivo en una ruta problemática o en un lugar sin permisos de escritura.

Ejemplo

python src/regulon_summary.py data/input.tsv /ruta/protegida/output.tsv

Qué se espera

  • El programa debe detectar el problema.
  • Debe informar que no pudo escribir el archivo.
  • No debe dejar un traceback crudo como única salida.

Qué se discute aquí

Este caso ayuda a recordar que un programa robusto no solo valida la entrada; también debe manejar adecuadamente la escritura de resultados.

Caso 4: el filtro --min_genes deja el resultado vacío

Qué se va a hacer

Ejecutar el programa con un valor alto en --min_genes para que ningún TF cumpla el filtro.

Ejemplo

python src/regulon_summary.py data/input.tsv results/output.tsv --min_genes 999

Qué se espera

  • El programa puede terminar normalmente.
  • Puede generar un archivo vacío o solo con encabezado.
  • Idealmente debe advertir al usuario que no hubo resultados.

Qué se discute aquí

Este caso es muy útil porque no necesariamente representa una excepción. Permite discutir la diferencia entre:

  • error real,
  • resultado válido pero vacío,
  • mensaje útil para interpretación.

Caso 5: el archivo tiene cero interacciones válidas

Qué se va a hacer

Usar un archivo donde todas las líneas sean descartadas por formato o por efecto inválido.

Qué se espera

  • El programa no debe romperse automáticamente.
  • Debe quedar claro que no hubo datos útiles para construir el regulón.

Qué se discute aquí

Este caso ayuda a pensar si conviene tratar esta situación como advertencia o como error, según el objetivo del programa.

💬 9. Discusión guiada con los estudiantes (MUY IMPORTANTE)

Esta sección no debe verse como un complemento menor. En esta lección, la discusión es central, porque el manejo de errores no consiste solo en aprender sintaxis (try, except, etc.), sino en tomar decisiones de diseño.

La meta de estas preguntas es que los estudiantes razonen sobre cómo debe comportarse un programa real frente a situaciones imperfectas.

9.1 Preguntas para abrir la discusión

  • ¿Un programa robusto es el que nunca falla, o el que falla de forma clara?
  • ¿Es mejor detener el programa ante cualquier problema, o intentar continuar en algunos casos?
  • ¿Qué diferencias hay entre un error del programador y un problema en los datos o en el entorno?

Propósito de estas preguntas

Sirven para mover la conversación de “cómo se escribe un except” hacia “qué comportamiento queremos diseñar”.

9.2 Preguntas sobre responsabilidades

  • ¿Dónde debe manejarse el error: dentro de la función o en main()?
  • Si load_interactions() encuentra un problema, ¿debe imprimir un mensaje o lanzar una excepción?
  • Si cada función imprime sus propios errores, ¿qué ventajas y desventajas tendría?

Qué se busca discutir

Aquí queremos que los estudiantes vean que el manejo de errores también se relaciona con organización del código y separación de responsabilidades.

Una función puede:

  • resolver su tarea,
  • detectar un problema,
  • reportarlo hacia arriba,

mientras que main() puede quedar como el punto donde se decide qué mostrar al usuario final.

9.3 Preguntas sobre líneas inválidas en el archivo

  • Si una línea tiene menos columnas de las esperadas, ¿qué hacemos?
  • Si effect no es +, - o +-, ¿la ignoramos o detenemos el programa?
  • ¿Conviene advertir cuántas líneas se descartaron?
  • ¿En qué escenarios tendría sentido un modo estricto?

Qué se busca discutir

Esta parte es muy rica porque no hay una sola respuesta correcta. Depende del objetivo del software.

Por ejemplo:

  • en un modo flexible, el programa puede ignorar líneas malas y seguir,
  • en un modo estricto, una línea inválida podría considerarse suficiente para detener el proceso.

Aquí el estudiante comienza a pensar como desarrollador, no solo como alguien que escribe instrucciones sueltas.

9.4 Preguntas sobre la experiencia del usuario

  • Si el programa falla, ¿qué información necesita realmente el usuario?
  • ¿Le sirve un traceback completo?
  • ¿Qué mensaje sería más útil para alguien que solo quiere ejecutar el programa?
  • ¿Qué detalles sí conviene mostrar y cuáles podrían ser excesivos?

Qué se busca discutir

Que un buen manejo de errores también mejora la experiencia de uso del software.

No se trata únicamente de “atrapar excepciones”, sino de comunicar bien:

  • qué falló,
  • dónde falló,
  • y, cuando sea posible, qué puede hacer el usuario.

9.5 Preguntas sobre resultados vacíos

  • Si no queda ningún TF después del filtro --min_genes, ¿eso es un error?
  • Si todas las líneas fueron descartadas, ¿el programa debería avisar?
  • ¿Es suficiente generar un archivo con encabezado y nada más?

Qué se busca discutir

Esta parte ayuda a distinguir entre:

  • una excepción real,
  • una advertencia,
  • y un resultado válido pero sin información útil.

Es una discusión muy importante porque muchas veces el software científico no “falla”, pero sí produce resultados difíciles de interpretar si no se comunican bien.

9.6 Preguntas sobre buenas prácticas

  • ¿Es buena idea usar except: sin especificar el tipo de error?
  • ¿Qué riesgo hay en capturar demasiado?
  • ¿Qué ventaja tiene capturar errores específicos como FileNotFoundError o PermissionError?

Qué se busca discutir

Que capturar errores específicos vuelve el código más claro y evita esconder problemas inesperados.

También permite mensajes más precisos y decisiones más adecuadas según el tipo de fallo.

9.7 Cierre de la discusión

Una forma útil de cerrar esta sección es pedir a los estudiantes que completen verbalmente o por escrito esta frase:

“Para nuestro programa, cuando ocurra ________, decidimos ________ porque ________.”

Ejemplos:

  • “Cuando una línea tenga un effect inválido, decidimos ignorarla porque el resto del archivo puede seguir siendo útil.”
  • “Cuando no se pueda escribir la salida, decidimos detener el programa porque el resultado no puede guardarse.”
  • “Cuando el filtro deje el resultado vacío, decidimos mostrar una advertencia porque no es una falla técnica, pero sí una situación relevante.”

Esta dinámica ayuda a que las decisiones de manejo de errores queden explícitas y justificadas.

🧠 10. Reflexión final

Manejar errores no es evitar que el programa falle…

Es decidir cómo debe comportarse cuando el mundo real no es perfecto.

🚀 11. Actividad práctica

Objetivo

Robustecer regulon_summary

Tareas

  1. Agregar manejo de errores en:

    • lectura
    • escritura

  2. Ejecutar casos de prueba

  3. Documentar decisiones:

  • ¿Qué errores decidiste capturar?
  • ¿Qué errores decidiste ignorar?
  • ¿Por qué?

🧪 12. Extensión (opcional)

  • Contar líneas inválidas.
  • Mostrar advertencias.
  • Modo --strict (detener ante errores).

📌 13. Idea clave de la lección

Un programa robusto no es el que nunca falla…

sino el que falla de forma clara, controlada y útil para el usuario.
